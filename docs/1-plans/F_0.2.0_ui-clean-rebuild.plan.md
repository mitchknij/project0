# UI Clean Rebuild Implementation Plan

**Version**: 0.2.0 (minor — full rework of the UI layer, no save/data impact)
**Date**: 2026-07-16
**Scope decisions (user-confirmed)**: clean rebuild of the `UIBuilder` output keeping the bake pipeline; drop ALL PNG art from code (pure procedural theme); everything in scope (title/character flow + HUD + all sub-panels); every panel keeps a HUD entry point.

## Overview

Rebuild the code-built uGUI layer so playtesting is trustworthy: one consistent procedural theme with zero PNG-art dependencies, shared layout constants instead of magic numbers, dead code and stale half-migration comments removed, and hardened runtime-vs-baked behavior. The proven pipeline stays: `UIBuilder.BuildAll` → `UIBakeTool` bake → `GameUI.prefab` (the Editor owns the final artifact).

## Problem Statement

The UI works but carries the residue of abandoned experiments (survey 2026-07-16):

- The PNG art path half-landed: `PreferSlicedArt = true` (`UITheme.cs:37`) loads `HUD/UI Bottom Main` but uses only the `hud_bar` slice (`UITheme.cs:155`); `slot_0..slot_5` are loaded and never referenced; `SkillIcon`/`skills.png` have zero callers; missing icons silently return `null`.
- Stale contradictory comments: `UIBuilder.cs:328/:386/:432` describe a layout ("Crafting tijdelijk verwijderd", "geen slot-knop voor Bank") that no longer matches the code — Bank and Craft buttons ARE created (`UIBuilder.cs:387-388`).
- Magic-number layout: HUD width `12*SlotWidth + 745f + 11*8f` (`UIBuilder.cs:313`) and scattered per-widget `preferredH` literals; only 4 shared constants exist (`UIBuilder.cs:294-297`).
- Fragile wiring: `MainHudPanel.BindButton` falls back to matching child GameObjects by name string (`MainHudPanel.cs:109`) — a rename breaks navigation silently.
- Runtime-vs-baked ambiguity: `UIBuilder.Bootstrap` suppresses the procedural build if ANY `Canvas` exists (`UIBuilder.cs:18`) — any stray Canvas silently kills the UI; a scene without the prefab silently diverges from the baked artifact.
- Dutch comments/log strings throughout (comments and Debug strings only; user-facing text is already English) plus the Dutch input placeholder default (`UIHelpers.cs:260`).

## Solution Architecture

Keep the three-stage architecture unchanged — **build code → bake → prefab** — and clean each stage:

1. **Theme (UITheme)** becomes procedural-only: palette, generated 9-slice frames (`PanelFrame`, `ButtonFrame`, `SlotFrame`, `InsetFrame`, `BarFrame`, `BarFill`), and the two pixel fonts (fonts stay — they are not the PNG art being dropped). All art-loading paths are deleted, not disabled, so nothing can half-load again. **No art files are deleted from disk** — `Assets/Art/UI/**` stays untouched for a future reskin; only code references go.
2. **Layout metrics centralize** into one shared constants block in `UITheme` (panel card sizes, HUD height, button/slot sizes, spacing, padding). HUD width becomes computed from its actual children count — no `745f`.
3. **Builder (UIBuilder)** is rewritten method-by-method against those constants: same panel inventory (all 12), same `BuildSubPanel<T>` pattern, text-labeled nav buttons for all 7 sub-panels (Inventory, Equipment, Activity, Travel, Crafting, Talents, Bank) + Auto toggle + 3 skill slots. All comments English.
4. **Wiring hardens**: nav binding uses only the direct button references `UIBuilder` already passes (name-string fallback removed). **UI ownership is made explicit**: the baked `GameUI.prefab` instance lives in `PersistentGame.unity` (which currently contains NO GameUI instance — verified 2026-07-16, `grep GameUI` → 0 hits). Because `UIBuilder.Bootstrap` runs at `AfterSceneLoad` of the FIRST scene (`Bootstrap.unity`) — i.e. BEFORE `SceneLoader.LoadInitialScenes` additively loads `PersistentGame` (`SceneLoader.cs:43-47`) — an immediate "is the UI here?" check would build a runtime UI and then receive the baked one: two UIs. The fallback therefore becomes **deferred**, with an explicit completion signal: `SceneLoader` gains a `public bool InitialLoadCompleted` (set `true` on every exit path of `LoadInitialScenes` — success, early-return, and failure — so the flag always resolves; one View-layer change, in scope). The fallback logic: if a `SceneLoader` exists, wait until `InitialLoadCompleted`; otherwise wait one frame; then build only if no `UIBootstrapper` exists anywhere. This keeps direct-Play of legacy scenes (`Scene A`) working while making the baked instance authoritative in the real boot flow. (UI may legally read `SceneLoader` — `IdleCloud.UI` already references `IdleCloud.View`.)
5. **Bake tool (UIBakeTool)** keeps its flow, with one gate fix: the current abort check skips persistent sprites entirely (`EditorUtility.IsPersistent(s) → continue`, `UIBakeTool.cs:161`), so a `Resources`-loaded PNG (persistent) passes silently — it is NOT a no-art regression gate today. The gate is extended to also reject any persistent `Image.sprite` whose asset path is outside `Assets/Art/UI/Generated/` (and any persistent TMP font outside `Generated/Fonts/`). Only then does "bake succeeds" prove no art reference survived.

**Current state**: palette + procedural frame generation + font pipeline + three art loaders (`HudBarSprite` sheet load at :146-155, `Icon` at :163, `SkillIcon` at :172-176) gated by `PreferSlicedArt` (:37).

**Modifications**:

- Add a `Layout` constants block (nested static class or const fields): HUD height, nav button width/height, skill-slot size, panel card sizes (title 520×420, char-select 640×560, char-create 700×660, offline 640×560 — current values kept), standard spacing (8), standard padding, bar heights. `UIBuilder`'s four consts (:294-297) move here.
- Translate remaining Dutch comments/log strings in this file.

### 2. UIHelpers — factory cleanup

**Current state**: complete widget factory; `CreateButton(parent, label, bgColor, frameSprite, iconSprite, iconSize)` supports icon-instead-of-text; Dutch placeholder default `"Voer tekst in..."` (:260).

**Modifications**:

### 3. UIBuilder — the rebuild proper

**File**: `Assets/Scripts/UI/UIBuilder.cs` (619 lines)

**Current state**: `BuildAll` (:27) + per-panel builders; `Bootstrap` `[RuntimeInitializeOnLoadMethod]` no-ops if any Canvas exists (:18); HUD builder (:299-439) mixes icon buttons, magic widths, stale comments.

**Modifications**:

- `Bootstrap` (:15-25): replace the immediate any-`Canvas` check with the **deferred fallback** (Solution Architecture item 4): if a `SceneLoader` exists, wait until `SceneLoader.InitialLoadCompleted`; otherwise wait one frame; then build only if no `UIBootstrapper` exists in any loaded scene. (The `RuntimeInitializeOnLoadMethod` itself stays; it just starts the deferred check — via a small runner coroutine host — instead of building synchronously.)
- `BuildMainHudPanel` (:299): rewrite — nav row of 7 text buttons (Inventory, Equipment, Activity, Travel, Crafting, Talents, Bank) + Auto toggle, center stats block unchanged in content (name/gold, HP/XP bars via `CreateBar`, map/activity), 3 skill slots with cooldown fills/hotkeys; frame = `PanelFrame`. Width is exactly defined: `HudWidth = 8·NavBtnW + CenterBlockW + 3·SlotSize + (11)·Spacing + 2·EdgePad` — every term a `UITheme.Layout` constant (8 = 7 nav + Auto; 12 children total → 11 gaps). `CenterBlockW` is a fixed constant with a separate `CenterBlockMinW`. Narrow-aspect policy (CanvasScaler match 0.5 shrinks the reference-space width below 1920 on narrower aspect ratios, which would clip a fixed-width HUD): the HUD root width is `min(HudWidth, current canvas width)`; the row's HorizontalLayoutGroup controls child widths, nav/skill buttons are fixed, and only the center block shrinks (between `CenterBlockW` and `CenterBlockMinW`) when clamped. At 16:9 and wider nothing shrinks.
- All `Build*` panel methods: restyle against `UITheme.Layout` constants, keep structure/behavior identical (each panel component's field wiring stays exactly as-is so `UIBootstrapper`/`GameFlowController` are unaffected).
- Remove every icon lookup (`UITheme.Icon(...)` call sites :329-333).
- Rewrite all comments in English; delete the stale "tijdelijk verwijderd" narrative (:328, :386, :432).

Renaming any of them breaks the test silently at runtime, not compile time.

### 5. UIBakeTool — bake pipeline sanity

**File**: `Assets/Editor/UIBakeTool.cs`

**Current state**: `ExportThemeAssets` writes 6 procedural frames + 2 SDF fonts to `Assets/Art/UI/Generated/`; `BakeUIPrefab` maps runtime sprites/fonts → exported assets and aborts on unmapped references (:173).

**Modifications**:

- Remove any mapping entries for the deleted art properties (HUD sheet / icons) so the sprite map is exactly the 6 procedural frames + fonts.
- **Extend the abort gate** (`:158-176`): today persistent sprites/fonts are skipped (`EditorUtility.IsPersistent → continue`), so a Resources-loaded PNG passes silently. Additionally fail the bake for any persistent `Image.sprite` outside `Assets/Art/UI/Generated/` and any persistent TMP font outside `Assets/Art/UI/Generated/Fonts/` (use `AssetDatabase.GetAssetPath`). Unity built-ins with empty/`Resources/unity_builtin_extra` paths: log-and-fail too — the rebuilt UI should not reference them. **Font-fallback boundary**: the TMP-default fallback in `UITheme` (:65-70) is a runtime-only safety net for direct-Play with missing fonts; at BAKE time it is not a valid state — a bake whose text resolved to the TMP default font (i.e. source fonts missing from `Resources/UI/Fonts`) intentionally aborts via this gate.
- Translate Dutch dialog/log strings.
- `PlaceGameUIInScene` places into the ACTIVE scene (`:212+`) — no code change, but the bake procedure (Phase 3) must have `PersistentGame.unity` open AND active, since it currently contains no GameUI instance and is the designated owner.

## Technical Considerations

- **Pattern Usage**: existing panel pattern preserved (component class + `UIBuilder` build method + `UIBootstrapper` registration + `PanelManager`); bake workflow per ARCHI.md §13 — the Editor owns the layout artifact.
- **Layer Placement**: changes in `IdleCloud.UI` asmdef + `Assets/Editor/` (Assembly-CSharp-Editor), plus ONE additive View change: `SceneLoader.InitialLoadCompleted` flag (`IdleCloud.View`). Direction stays legal — UI already references View; no layer gains a new reference. Zero Core/Data/Managers changes.
- **Determinism**: N/A — no simulation code touched.
- **Editor Authoring Impact**: after implementation the user must open `PersistentGame.unity` and make it the **active** scene, run `IdleCloud > UI > Bake UI Prefab` (regenerates `Assets/Art/UI/Generated/` + `GameUI.prefab`, places the instance into the active scene), verify the `GameUI` instance landed in `PersistentGame` (it currently has none — verified), save that scene, and run the Play checklist below. `Bootstrap.unity`/map scenes must NOT contain a GameUI instance.
- **Save Compatibility**: none — no snapshot/state shape changes.
- **Edge Cases**: scene with a stray non-game Canvas (new Bootstrap guard must still build); missing font assets (existing TMP fallback keeps working); baked-prefab scene vs empty scene (both paths must produce the same UI); `GameUI.prefab` overwrite must not orphan the `Generated/` GUIDs (bake tool already reuses assets to preserve GUIDs — keep that behavior).
