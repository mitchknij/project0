# Tile-First Targeting Implementation Plan v2

> **Authority:** `docs/guardrails/PROJECT.md` remains authoritative. This plan extends the implemented skill engine, `ActiveSim`, slot-order auto-combat, `GameSession`, existing ISO spatial data, scheduler, effects, events and single reward path. Do not create parallel combat, targeting, spatial, timing, state, UI or reward systems.

## 1. Overview

Make tile patterns the primary spatial language for new area skills.

A tile skill declares:

- how its anchor tile is resolved;
- which relative tile pattern it affects;
- which floor policy applies;
- which actors on those tiles are eligible;
- whether target-count conditions permit automatic use.

Core resolves affected tiles and actors deterministically from tile coordinates already present in combat spatial snapshots. Existing actor-targeted and continuous-ground skills remain unchanged.

The first vertical slices are:

1. **Ground Smash**, a self-anchored Cross;
2. **Arcane Detonation**, a target-anchored SquareRadius 1.

These prove both source-tile and target-tile anchoring without requiring authoritative facing or manual tile selection.

---

## 2. Problem statement

Current area skills primarily resolve through continuous world-space circles. That remains valid for existing skills, but circle boundaries do not naturally match the tile grid presented to the player.

The current combat spatial snapshot already carries actor tile and floor information. Tile-first targeting should consume that existing data rather than introducing another grid or position source.

The offline evaluator also needs to recognise tile-based area skills explicitly so they are not omitted or treated as continuous circles by accident.

---

## 3. Core architecture

The tile-first execution flow is:

```text
Existing skill command
→ resolve anchor tile from source or current target
→ generate an ordered affected-tile pattern
→ resolve unique eligible actors occupying those tiles
→ pass actors into the existing effect pipeline
→ reuse existing damage, crit, status, death and reward handling
→ emit the resolved tile pattern for presentation
```

Design rules:

- The existing ISO grid and `CombatSpatialAdapter` remain the only tile authority.
- Core never performs its own world-to-tile or tile-to-world conversion.
- Tile targeting determines where a skill acts. Existing skill systems still determine cooldowns, timing, slot priority, effects, statuses and rewards.
- An actor is hit at most once per skill resolution unless a future skill explicitly defines one hit per tile.
- Existing actor-targeted and continuous-circle skills remain regression references.
- Directional and manually selected tile patterns remain absent until their authoritative inputs exist.

## 4. Data contracts

Extend the existing flat targeting model rather than introducing a parallel targeting hierarchy.

### 4.1 Skill targeting kinds

Add only:

```text
SkillTargetingKind
├── TilePatternAroundSource
└── TilePatternAroundTarget
```

Do not add selected-tile or directional kinds in this release.

### 4.2 Tile pattern kinds

```text
TilePatternKind
├── SingleTile
├── Cross
├── SquareRadius
└── CustomOffsets
```

Definitions:

```text
SingleTile
    Anchor tile only

Cross(size)
    Anchor plus cardinal offsets up to the configured arm length

SquareRadius(radius)
    All offsets where max(abs(x), abs(y)) <= radius
    SquareRadius 1 therefore produces a 3 × 3 pattern

CustomOffsets
    Explicit ordered relative tile offsets supplied by content
```

### 4.3 TilePatternDef

Add one optional `TilePatternDef` to `ClassSkillDef`, beside existing continuous-targeting fields.

It contains only tile-specific configuration:

```text
PatternKind
Size
CustomOffsets
FloorPolicy
```

`RadiusWorldUnits` remains for continuous skills and is ignored for tile-targeting kinds.

### 4.4 Maximum targets

Keep `MaxTargets` with the existing action or skill targeting configuration if the current continuous system already owns target limits there. Do not give tile skills a different ownership convention merely because they use patterns.

If the current model has no shared target-limit field, Fable may keep `MaxTargets` in `TilePatternDef` for this release, but must document that decision and avoid creating two conflicting target caps.

### 4.5 Authoring validity

Content validation must enforce:

- tile-targeting kinds require a non-null `TilePatternDef`;
- non-tile targeting kinds must not carry active tile-pattern data;
- Cross and SquareRadius require `Size >= 1`;
- CustomOffsets must be non-empty;
- duplicate CustomOffsets are invalid authoring data and must be rejected;
- offset magnitudes must remain within a documented safe bound;
- unsupported floor or rotation policies are rejected;
- existing continuous skills remain valid without changes.

Core should still defensively guarantee unique resolved tiles and actors if malformed data somehow bypasses validation. Defensive deduplication is invariant protection, not supported authoring behaviour.

## 5. TilePatternResolver

Create one pure, engine-free `TilePatternResolver` beside the existing circle resolver.

Its responsibilities are:

```text
ResolveTiles(anchor, pattern)
→ ordered unique tile coordinates

ResolveActors(tiles, anchorFloor, sourceFaction, candidates)
→ ordered unique eligible actors
```

### 5.1 Canonical tile ordering

Ordering must be documented and covered by tests.

Use:

```text
SingleTile:
    Anchor

Cross:
    Anchor
    then each distance ring from 1 through Size
    within each ring: North, East, South, West

SquareRadius:
    Anchor first
    then deterministic row-major scan of remaining offsets
    rows ordered from negative Y to positive Y
    columns ordered from negative X to positive X

CustomOffsets:
    Declaration order
```

If the project uses a different cardinal-axis naming convention, preserve the numeric offset order while adapting labels to the existing grid.

### 5.2 Actor resolution

Initial hit policy:

> An actor is eligible when its authoritative occupied tile belongs to the affected pattern.

Then apply:

- compatible floor;
- opposing faction where required;
- alive state;
- targetable state;
- unique actor ID;
- target cap, if configured.

Actor ordering is:

```text
Affected tile order
→ stable runtime ActorId within each tile
```

This logical occupied-tile rule is deliberate. Do not mix in continuous footprint overlap for the first tile-first release.

The API may later support multi-tile actors, but do not implement multi-tile occupancy before an actual large enemy requires it.

## 6. ActiveSim integration

Extend the existing target-resolution paths for:

- immediate actions;
- scheduled impacts;
- automatic target-condition checks.

### 6.1 Anchor policies

```text
TilePatternAroundSource
→ source actor tile at resolution time, unless timing policy captures it earlier

TilePatternAroundTarget
→ current target tile, or captured target tile for scheduled impacts according to existing timing semantics
```

No valid anchor returns an empty result. The caller then applies normal eligibility, cancellation or fizzle behaviour.

### 6.2 Shared effect pipeline

Tile-resolved actor IDs enter the same processing used by current area skills:

- hit and crit resolution;
- damage and mitigation;
- statuses and scheduled effects;
- defeated-state handling;
- the existing single kill/reward transaction.

Do not create tile-specific damage or reward logic.

### 6.3 Automatic use

The current skill-bar policy remains unchanged:

```text
Slot 1
→ Slot 2
→ Slot 3
→ Slot 4
→ basic attack fallback
```

Target-pattern conditions affect eligibility only. They never alter slot priority.

For example, Ground Smash in Slot 1 is skipped when fewer than its configured minimum number of enemies occupy the affected Cross. The selector then checks Slot 2.

## 7. Combat events and presentation

Add a `TileAreaResolved` event containing:

```text
Affected ordered tile coordinates
Floor
Spatially resolved actor IDs
```

Event semantics must be explicit:

> `TileAreaResolved` describes the spatial resolution result. Individual effect events such as `DamageApplied` remain authoritative for actual damage or status outcomes.

A spatially resolved actor may later miss, resist or receive a non-damage effect. Presentation must not treat every resolved actor ID as proof of damage.

### 7.1 Tile-to-world conversion

`CombatFeedbackView` must not discover or assume an arbitrary scene `Grid`.

Use one explicit conversion seam:

```text
Existing authoritative ISO tile-to-world conversion service
or
an explicitly serialized/injected reference to the same scene Grid used by world authoring
```

The conversion must match the conventions that produced the actor tile snapshots, including:

- grid origin;
- cell anchor;
- tile scale;
- floor or elevation handling.

Core emits tile coordinates only. View converts them for presentation through the existing world adapter and does not recreate tile geometry.

### 7.2 Placeholder highlight

For each affected tile, show a short-lived placeholder overlay with Inspector-configurable:

- tint;
- duration;
- optional element color;
- render offset required by the existing ISO presentation.

The View renders the exact Core-provided tile list. Visual overlays never determine hits.

## 8. Starter skills

### 8.1 Ground Smash

```text
Targeting: TilePatternAroundSource
Pattern: Cross
Size: 1
Tiles: anchor plus four cardinal neighbours
Effect: Physical area damage
Timing: immediate
Auto-use condition: MinimumAutoTargets = 2 initially
```

The value `2` is provisional but deliberate: Ground Smash should initially demonstrate meaningful multi-target gating rather than replacing single-target attacks whenever one enemy is nearby.

A later balance change may allow use against one target, but that must be an explicit content decision.

### 8.2 Arcane Detonation

```text
Targeting: TilePatternAroundTarget
Pattern: SquareRadius
Size: 1
Tiles: 3 × 3 around current target tile
Effect: Arcane area damage
Timing: immediate initially
Auto-use condition: valid current target
```

Initial damage and cooldown values remain content-tuning decisions consistent with existing class balance.

### 8.3 Follow-up skills, not part of this release

After these two skills prove the resolver:

- Flame Patch may add target-tile status or persistent-tile behaviour;
- Frost Field may add tile AoE plus Chill;
- Tile Barrage may add repeated scheduled patterns;
- directional Tile Line waits for authoritative direction;
- selected-tile Meteor waits for authoritative tile-selection input.

Do not add those contracts or dead enum members now.

## 9. Relationship with existing skills

Keep current behaviour unchanged:

```text
Power Strike   → actor-targeted
Arcane Bolt    → actor-targeted scheduled projectile
Guard          → self-targeted modifier
Flame Burst    → existing target-centred continuous skill
Earthbreaker   → continuous-circle comparison reference
```

After Ground Smash and Arcane Detonation are accepted in Scene A, decide separately whether any existing circle skill should remain, be replaced or gain a tile-based variant.

Do not silently rebalance or convert existing skills during this implementation.

## 10. Offline approximation

Tile skills must participate in the existing 60-second slot-order approximation without per-kill simulation.

The affected tile count is a **hard upper bound**, not automatically the expected number of targets.

Use a conservative documented estimate derived from the existing encounter-density model:

```text
ExpectedTargetsPerCast = min(
    affected tile count,
    density-derived expected occupied tiles for this pattern,
    optional skill-specific conservative cap
)
```

Requirements:

- reuse current slot order;
- respect cooldowns and normal eligibility assumptions;
- do not assume every affected tile contains an enemy;
- exclude a skill when its expected valid target count does not meet `MinimumAutoTargets`;
- retain analytical, rate-based complexity;
- record the pattern and density assumption in `AssumptionSource`;
- do not optimise the rotation beyond the player-configured skill-bar order.

The exact density conversion should be chosen after inspecting the existing `Activity.cs` model. Do not copy a world-space circle assumption without verifying its units and meaning.

## 12. Implementation phases

### Phase 1 — Contracts and pure resolver

- add the two targeting kinds;
- add `TilePatternKind` and `TilePatternDef`;
- add the optional pattern to `ClassSkillDef`;
- implement validation;
- implement canonical tile generation and actor resolution;
- perform switch exhaustiveness sweeps;
- build and run focused Core tests.

### Phase 2 — Engine integration and starter content

- integrate immediate and scheduled tile-resolution paths;
- integrate auto-combat eligibility;
- add `TileAreaResolved` with clear spatial-event semantics;
- add Ground Smash and Arcane Detonation;
- preserve the existing reward path;
- add tile skills to offline approximation.

### Phase 3 — Presentation and acceptance

- connect explicit tile-to-world presentation conversion;
- render placeholder tile highlights;
- run all automated regression tests;
- manually verify Ground Smash and Arcane Detonation in Scene A or the current playable map.

## 13. Validation and tests

### Automated tests

Cover:

- SingleTile output;
- Cross size and canonical order;
- SquareRadius as Chebyshev distance and canonical order;
- CustomOffsets declaration order;
- duplicate CustomOffsets rejected by validation;
- defensive resolver deduplication;
- same-floor filtering;
- alive, targetable and faction filtering;
- actors ordered by tile then ActorId;
- target cap applied after deterministic ordering;
- actor hit only once;
- missing source or target anchor;
- Ground Smash minimum-target eligibility;
- Slot 1 skipped when pattern condition fails;
- basic-attack fallback;
- Arcane Detonation target anchoring;
- scheduled impact using the intended captured anchor semantics;
- `TileAreaResolved` payload matching the spatially resolved set;
- actual damage represented through the existing effect events;
- tile-skill kill using the single reward transaction;
- offline expected-target upper bound and documented density assumption;
- all existing continuous-skill regression tests unchanged.

### Manual acceptance

Verify:

1. Ground Smash hits enemies on the player tile and four cardinal neighbours.
2. Ground Smash does not auto-cast against one enemy when `MinimumAutoTargets = 2`.
3. An enemy visually nearby but on an unaffected tile is not hit.
4. An enemy on an incompatible floor is not hit.
5. Reordering Ground Smash in the skill bar changes its automatic priority.
6. Arcane Detonation anchors around the selected slime's logical tile.
7. Highlighted tiles exactly match the Core event payload.
8. Damage popups and actual affected actors remain consistent with normal hit, crit or resistance outcomes.
9. Existing Earthbreaker, Power Strike and Arcane Bolt behaviour is unchanged.

---

## 14. Definition of done

The tile-first foundation is complete when:

- the existing ISO grid remains the only tile source of truth;
- the flat `SkillTargetingKind` cleanly supports source- and target-anchored tile patterns;
- `TilePatternDef` contains patterns without duplicating existing action fields;
- pattern semantics and ordering are documented;
- duplicate authoring offsets are rejected;
- Core defensively returns unique tiles and actors;
- tile actors resolve through the existing effect and reward pipeline;
- `TileAreaResolved` is spatially descriptive and does not replace effect events;
- View uses the authoritative existing tile-to-world conversion seam;
- Ground Smash and Arcane Detonation work end to end;
- Ground Smash initially requires two automatic targets;
- offline expected targets use a conservative density-derived estimate rather than raw tile count;
- existing actor-targeted and continuous skills remain unchanged;
- automated tests and manual acceptance pass.
