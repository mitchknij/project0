# Mining Lifeskill on FirstMap Copper Rocks — Implementation Plan (v0.7.0)

## Overview

Make the Mining lifeskill playable on the copper rocks the user placed in FirstMap: clicking a rock walks the player to it and mines it in repeating swings, a world-space progress bar on the rock fills over each swing, successful swings drop `copper_ore` into the inventory/bank, and the ore feeds the existing copper crafting recipes. Almost the entire backend already exists (`copper_vein` in `NodesRepo`, `LifeSkills.Tick`, `ActiveGatheringCoordinator`, `GatheringView`/`GatheringNodeView`, `CraftingPanel` + copper recipes); this feature closes the last mile: an interactable Rock prefab in the scene, per-swing progress exposed by Core, and the bar visual.

## Problem Statement

- **No gathering node exists in any scene.** `GatheringView` is placed in FirstMap, but zero `GatheringNodeView` components exist in any `.unity` file — gathering has never been live. The user's rocks are plain `Rock_1` sprite objects (from the sliced `Assets/Art/Tilesets/Rock.png`) with no collider, no sort controller, and no node identity; the scene changes are not yet saved to disk.
- **Core exposes no per-swing progress.** `LifeSkills.Tick` computes the swing interval internally and only reports completed actions; the View has no data to render a filling bar.
- **No world-space progress visual exists** for gathering nodes (nearest pattern: the enemy health bar in `CombatTargetView`).
- **Crafting can't complete with ore alone**: both copper recipes also require `oak_log`, and trees are likewise not yet gatherable (no `GatheringNodeView` anywhere). One tree must be wired the same way or the "craft equipment" goal is unreachable.
- **Crafting consumes bank materials and coins only** (`Crafting.CanCraft`/`Craft` operate on `Bank`; pickaxe costs 30 coins, sword 50). New accounts start with `Coins = 0` and `AutoLoot = false`, so gathered items land in the character inventory, not the bank. The verification path must earn coins (monster kills drop 1–3) and get materials into the bank (enable Auto Loot or deposit via the Bank panel).

User decisions (recorded 2026-07-18): per-swing progress bar (no depletion/respawn), no pickaxe requirement (level gate only), existing recipes suffice, rocks are already placed as objects in the live Editor.

## Solution Architecture

Three thin slices, respecting Data ← Core ← Managers ← View:

1. **Core/Data — expose swing progress.** `ActiveGatheringResult` gains two runtime-only fields (swing interval ms + progress fraction 0..1). `LifeSkills.Tick` populates them on the active in-range path from `state.LastResolvedAt` and the already-computed interval — pure arithmetic, no RNG, deterministic. Additionally, while the target is out of range, `state.LastResolvedAt` is advanced to the current timestamp so walking time does not accrue into instant swings on arrival (today a long walk would complete swings retroactively; the bar would make that visible as a broken-looking jump).
2. **View — bar on the node + correct node instance.** `GatheringNodeView` grows a hidden two-sprite world-space bar (same construction pattern as `CombatTargetView.CreateHealthBar`: runtime 1-px white sprite, background + fill children, Inspector-tunable size/colors) with a small show/set/hide API. `GatheringView` remembers the clicked node instance (several rocks share the `copper_vein` definition id) and prefers it when ticking, feeds the progress fraction into the bar each tick, and hides the bar on stop/idle/out-of-range/target switch.
3. **Editor authoring (user, live Editor — no code).** Generate `Rock_*` prefabs with the existing `WorldAssetPrefabGenerator` using `Tree_Base.prefab` as the base profile (SortingGroup + `WorldObjectSortController` + shadow + lit Visual), add `GatheringNodeView` (`nodeId: copper_vein`) to the used Rock prefab, replace the placed plain sprites with prefab instances, wire one Tree prefab with `nodeId: oak_tree` the same way, save the scene, Play-verify.

## Implementation Details

### 1. `Assets/Scripts/Data/State/ActiveGatheringContracts.cs` (Data/State, modify)

**Current state**: `ActiveGatheringResult` carries `State`, `Events`, `SuccessfulActions`, `SkillXp`, `Loot` (lines 65–73). Runtime-only contract; never serialized into saves.

**Modifications**:
- Add two fields to `ActiveGatheringResult`: the effective swing interval in ms (`long`, 0 when not actively gathering) and the elapsed fraction of the current swing (`double` 0..1, 0 when not actively gathering). Names finalized in TRIP-2; something like `ActionIntervalMs` / `ActionProgress01`.

### 2. `Assets/Scripts/Core/LifeSkills/LifeSkills.cs` (Core, modify)

**Current state**: `Tick` gates on commands → active → target match → availability → range → skill level, then computes `interval` (line 40) and resolves whole `actions`; leftover partial progress is implicit in `state.LastResolvedAt`.

**Modifications**:
- Out-of-range branch (around line 26–30): advance `state.LastResolvedAt` to `input.Timestamp` before returning, so no swing time accrues while walking to the node (kills the retroactive-swing catch-up).
- Active in-range path (after resolving whole actions, around line 41–44): populate the two new result fields — interval, and `(Timestamp − LastResolvedAt) / interval` clamped to [0, 1]. Populated even when `actions == 0` (that is the common per-frame case the bar renders).

### 3. `Assets/Scripts/View/GatheringNodeView.cs` (View, modify)

**Current state**: identity + `Select()` only; `RequireComponent(CircleCollider2D)`; `Awake` looks up `GetComponent<SpriteRenderer>()` on its own GameObject (null on the Tree_Base-profile prefabs where the renderer sits on the `Visual` child — currently harmless).

**Modifications**:
- Build a hidden world-space progress bar in `Awake`, copying the `CombatTargetView.CreateHealthBar` construction (1-px shared sprite, background + fill child SpriteRenderers, sorting layer/order taken from the node's renderer). Measure the sprite top via `GetComponentInChildren<SpriteRenderer>()` (not `GetComponent`) so the Visual-child profile works.
- `[SerializeField]` tuning: bar width/height/vertical gap and background/fill colors (fill default: a mining-ish gold/amber to distinguish from the red enemy health bar).
- Public API: set progress (shows bar, resizes fill left-anchored) and hide. No Update loop of its own — `GatheringView` drives it.

### 4. `Assets/Scripts/View/GatheringView.cs` (View, modify)

**Current state**: `Update` finds the ticked node by **definition id** (`candidate.NodeId == character.Activity.TargetId`, line 59) — with five identical rocks it may tick/walk to a different rock than the one clicked. `HandleNodeSelected` forwards the click but discards the instance.

**Modifications**:
- Remember the clicked `GatheringNodeView` instance in `HandleNodeSelected`; in `Update`, prefer that instance when it is alive, available, and its `NodeId` matches the activity target; otherwise fall back to the existing first-match search. Clear it on stop/manual move.
- After each tick, drive the bar on the ticked node from the simulation's progress fraction; hide the bar (on the previously ticked node) whenever the activity is not a harvest, the node changed, the tick was rejected, or movement was requested (out of range).

## Technical Considerations

- **Pattern Usage**: bar construction copies `CombatTargetView.CreateHealthBar` (the repo's one existing world-space bar); prefab authoring reuses `WorldAssetPrefabGenerator` unchanged — `Tree_Base.prefab` is already a generic world-object profile and the window's Base Prefab / Source / Target fields are user-configurable. No editor-code changes.
- **Layer Placement**: contract fields in Data/State (`IdleCloud.Core` namespace, Data asmdef — engine-free ✓), progress math in Core (engine-free ✓), all rendering in View. No Managers change: `ActiveGatheringCoordinator` already passes the full `ActiveGatheringResult` through to the View.
- **Determinism**: progress fields are pure timestamp arithmetic, no RNG. The out-of-range `LastResolvedAt` advance is deterministic and does not touch offline math (offline uses rate-based `Activity`/`Offline`, not `ActiveGatheringState`).
- **Editor Authoring Impact** (user, live Editor — the whole feature is inert without these): prefab generation, adding `GatheringNodeView` to prefabs, scene replacement, scene save, Play-mode verify. Detailed in To-dos Phase 2. No UI rebake (no uGUI panel changes).
- **Save Compatibility**: none. `ActiveGatheringResult` is runtime-only; `ActiveGatheringState`'s shape is unchanged; no save fields added.
- **Edge Cases**:
  - Five rocks share `copper_vein`: clicked-instance tracking keeps walk target, ticks, and bar on the rock the user clicked; definition-id fallback preserved.
  - Walk-then-mine: no progress accrues while out of range (Core change above); bar starts at 0 on arrival.
  - Bar fraction clamped [0,1] against clock jitter; bar hidden when node unavailable or activity stops.
  - No depletion/respawn: rocks stay mineable forever (user decision); `IsAvailable` remains `activeInHierarchy`.
  - `WorldMapContext` default `DiscoverAuthoredComponents` mode auto-finds placed `GatheringNodeView`s — no explicit list curation needed; its `OnValidate` already warns on unknown/wrong-map node ids.

## Backward Compatibility

- New result fields default to 0; existing consumers (`ActiveGatheringCoordinator`, tests) ignore them — additive change.
- Behavior change: out-of-range time no longer converts into retroactive swings on arrival. This is a deliberate active-play fix; offline gains are unaffected (separate rate-based path). One existing test could rely on the old catch-up behavior — TRIP-2 must check `LifeSkillsTests` before editing.

## Scope Change (user-approved 2026-07-19, during Phase 3 verification)

Crafting must consume materials from the **bank AND the selected character's inventory** (user hit the bank-only wall with 9 ore in inventory and chose the design change over depositing). Decisions:

- `Crafting.CanCraft`/`Craft` (Core) gain the character: materials are counted/consumed across bank + inventory, **bank first, then inventory** (`Inventory.RemoveFromInventory`); **coins stay bank-only** (shared account currency); crafted output goes to the **character inventory** ("Inventory is full" throws — second user amendment, 2026-07-19). `Craft` returns updated bank + character.
- Call sites: `GameManager.CraftRecipe` (commit updated bank and character; session event must refresh both bank and inventory UI) and `CraftingPanel.Refresh` (pass the character).
- Diverges deliberately from store.ts crafting — recorded here as the authoritative decision.
