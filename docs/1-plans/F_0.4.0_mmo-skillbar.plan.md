# MMO Skillbar Implementation Plan (v0.4.0)

## Overview

Replace the HUD's three fixed, auto-populated skill buttons with a standalone MMORPG-style skillbar: 8 slots, positioned independently of the main HUD (bottom-center, above the HUD bar), with hotkeys 1–8. Players assign skills by dragging them from a new **Skills panel** (a HUD sub-panel like Inventory/Talents) onto bar slots, rearrange by dragging between slots, and clear by dragging a skill off the bar. Slot assignments persist per character in the save file.

User decisions (2026-07-16, via AskUserQuestion):
- 8 slots, skillbar separate from the current HUD; **remove** the current 3 skill buttons on the HUD's right side.
- Drag source = new Skills panel.
- Persist per character (save-format change accepted).
- All three drag interactions: assign from panel, swap between slots, drag-off to clear.

Noted assumptions (not blocking, flagged for user review):
- Bar placement: bottom-center, anchored above the HUD bar — exact position/size is Editor-tunable after the bake.
- **Auto-combat is unaffected**: `AutoCombatPolicy` continues to select skills by `ClassSkillDef.Priority` from the full class skill list, independent of what's on the bar. The bar governs manual triggering only.
- A skill may occupy at most one slot: assigning a skill that already sits in another slot *moves* it (old slot cleared) — standard MMORPG behavior.
- No icon art: `UITheme` is procedural-only (v0.2.0 rule, enforced by the bake gate). Slots and drag ghosts render the skill name text tinted by element/`AoeColor`, same as today's buttons.

## Problem Statement

Today `MainHudPanel` shows exactly 3 skill buttons whose contents are derived (`SkillAtSlot`) from `ClassDef.Skills` sorted by `Priority` — the player has no control over which skill sits where, there is no room for more than 3 skills, and the buttons are squeezed into the HUD bar. There is no skill browser UI at all: skills exist only as data in `ClassesRepo`.

## Solution Architecture

Four cooperating pieces, respecting the Data ← Core ← Managers ← UI flow:

1. **Data**: `Character` gains a `SkillBar` field — a fixed-length (8) list of skill-id strings, `null` = empty slot. Pure state, engine-free, Newtonsoft-serialized like every other character field.
2. **Managers**: `GameManager` exposes the only mutation API (`AssignSkillToBar` / `SwapSkillBarSlots` / `ClearSkillBarSlot`) so UI never writes state directly; `SaveManager` bumps `SaveSchemaVersion` 2 → 3 with a migration that seeds the bar from today's priority-derived layout (slots 0–2), so existing players see no change on first load.
3. **UI — SkillBarPanel** (new root under the GameUI canvas, sibling of `MainHudPanel`): renders the 8 slots, owns hotkeys 1–8 (taken over from `MainHudPanel`), cooldown fills, and click-to-trigger (unchanged path: `GameManager.EnqueueCombatCommand` with `CombatCommandKind.TriggerSkill`).
4. **UI — SkillsPanel** (new sub-panel, opened by a new HUD nav button): lists all `ClassDef.Skills` of the selected character with name/description/cooldown; each entry is a drag source. Drag-and-drop uses standard uGUI `IBeginDragHandler`/`IDragHandler`/`IEndDragHandler`/`IDropHandler` interfaces through the existing sole EventSystem (PersistentGame).

Trigger flow (Flow A) is untouched: bar slot click/hotkey → `CombatCommand` → `ActiveSim`. Core needs no changes for triggering; only `StateInvariantValidator` learns the new field.

## Implementation Details

### 1. Character state — `SkillBar`

**File**: `Assets/Scripts/Data/State/GameTypes.cs`

**Current state**: `Character` (partial class, ~line 192) has no skill-loadout field; a second partial (~line 281) implements `Clone()` with an explicit copy per field.

**Modifications**:
- Add `public List<string> SkillBar;` to `Character` — semantic contract: length exactly `SkillBarSlots` (8), entry = `ClassSkillDef.Id` or `null`. Document the contract in a doc-comment.
- Add a shared constant for the slot count in Data (e.g. a small static on/near `Character` or a `GameConstants` holder — implementer's choice, but it must live in Data so Core validation and UI both read the same number without duplicating `8`).
- Extend `Character.Clone()` (~line 288) to deep-copy the list (fresh `List<string>`; same pattern as `UnlockedWaypoints` in `Account.Clone()`).

### 2. New-character creation

**File**: `Assets/Scripts/Core/Character.cs`

**Current state**: `CharacterHelper.CreateCharacter` (line 45) builds a fresh level-1 character; it does not know about `SkillBar`.

**Modifications**:
- Initialize `SkillBar` to a length-8 list seeded with the class's skills sorted ascending by `Priority` into the leading slots, remainder `null` (same seeding rule as the migration below — extract one shared helper so the rule exists once; Core may read `ClassesRepo` since Core references Data). New characters thus start with a populated bar, matching what migrated characters see.

### 3. State validation

**File**: `Assets/Scripts/Core/Common/StateInvariantValidator.cs`

**Modifications**:
- Validate per character: `SkillBar` is non-null and length-8; every non-null entry matches a skill id in the character's `ClassDef.Skills`; no duplicate non-null entries. Follow the file's existing report/violation style. (Report-only, as today — actual repair happens in the SaveManager normalization pass below.)

### 4. Save migration (schema v2 → v3) + load normalization

**File**: `Assets/Scripts/Managers/SaveManager.cs`

**Current state**: `CurrentSaveSchemaVersion = 2` (line 35); `MigrateLoadedData` (line 156) applies ordered `if (sourceSchemaVersion < N)` steps, followed by unconditional repair passes like `MigrateEfficiencySnapshots`; `BackupSaveBeforeMigration` writes a `.pre-vN.bak` **only under `#if UNITY_EDITOR || DEVELOPMENT_BUILD`** (line 219).

**Modifications**:
- Bump `CurrentSaveSchemaVersion` to 3.
- Add a `< 3` migration step: for each character with `SkillBar == null`, seed via the shared helper from §2 (priority-sorted class skills into leading slots) — old saves reproduce today's visible layout exactly.
- Add an **unconditional, idempotent normalization pass** (runs on every load, version-independent, same placement as `MigrateEfficiencySnapshots`): enforce length exactly 8 (pad with `null` / truncate), drop ids not present in the character's `ClassDef.Skills` (content drift), and null-out duplicate occurrences after the first. This is the repair authority; the Core validator only reports. Covers malformed v3 saves and hand-edited files, not just the v2→v3 hop.
- Backup note: the automatic pre-migration backup exists only in Editor/development builds — acceptable while no release pipeline exists (ARCHI §19); do not treat it as a player-facing guarantee.

### 5. Skillbar mutation API

**File**: `Assets/Scripts/Managers/GameManager.cs`

**Current state**: `GameManager` already owns character mutations and exposes `EnqueueCombatCommand`, `ToggleAutoCombat`, etc.; UI never writes `Character` fields directly.

**Modifications**:
- Add three public methods operating on the selected character: assign (skill id → slot index, relocating the id if it already occupies another slot), swap (two slot indices, works for empty slots too), clear (slot index). Each validates inputs (slot range, skill id belongs to the character's class), then mutates via the established pattern: `AccountHelper.UpdateCharacter(Account, SelectedCharacterId, ...)` → `CommitAccount(updated, new CharacterMutationCommand("assign-skillbar-slot"/..., SelectedCharacterId), SessionEventKind.CharacterChanged)` (same shape as equip/unequip at GameManager.cs:715/733).
- UI re-render needs no new bus: `SkillBarPanel` registers with the existing `UIRefreshDriver` polling path (like the HUD), which picks up the committed change.

### 6. SkillBarPanel (new)

**File**: `Assets/Scripts/UI/SkillBarPanel.cs` (new)

- Plain component + `IPanelView`-style Show/Hide, mirroring `MainHudPanel` conventions: `[HideInInspector]` public reference arrays (`slotButtons`, `slotCooldownFills`, `slotHotkeyLabels`, `slotNameLabels`, and the `SkillBarSlot` components), populated by `UIBuilder` at build time and baked into the prefab.
- Behavior moved from `MainHudPanel`: hotkeys (now digits 1–8, via `Keyboard.current` in `Update`), `TriggerSkillSlot` (resolves the slot's skill id from `Character.SkillBar` instead of `SkillAtSlot`, then `EnqueueCombatCommand`), cooldown fill refresh (same `SkillNextReadyAt` math, keyed by the slot's assigned skill id), interactable/label refresh.
- Visibility: `MainHudPanel.Show/Hide` also shows/hides the skillbar (HUD holds a baked reference to `SkillBarPanel`), so every existing flow-controller call site works unchanged.

### 7. SkillBarSlot + drag infrastructure (new)

**Files**: `Assets/Scripts/UI/SkillBarSlot.cs` (new), `Assets/Scripts/UI/SkillDragController.cs` (new)

- `SkillBarSlot`: per-slot component holding its slot index; implements `IDropHandler` (accept payload → `GameManager.AssignSkillToBar` or `SwapSkillBarSlots` depending on payload source) and `IBeginDragHandler`/`IDragHandler`/`IEndDragHandler` (dragging an occupied slot; ending a drag over no slot → `ClearSkillBarSlot`). Click-without-drag still triggers the skill (uGUI Button + drag threshold handles this natively; verify no click/drag conflict during Play check).
- `SkillDragController`: one canvas-level helper owning the drag ghost (a single reusable TMP label + frame, `raycastTarget = false`) and the current drag payload (`skillId` + source kind: panel-entry vs bar-slot + source slot index). Lives on the GameUI canvas, wired by `UIBuilder`. Keeping the payload here avoids static mutable state and keeps both drag sources symmetric.
- Layer note: all of this is pure UI (`IdleCloud.UI` asmdef, references UnityEngine.EventSystems — already available via uGUI).

### 8. SkillsPanel (new)

**File**: `Assets/Scripts/UI/SkillsPanel.cs` (new)

- Built with the existing `BuildSubPanel<T>` generic modal pattern in `UIBuilder` (same skeleton as `TalentsPanel`/`CraftingPanel`).
- Content: one row per `ClassDef.Skills` entry — name (element-tinted), description, cooldown seconds, mechanic. Each row hosts a drag-source component (can reuse `SkillBarSlot`'s drag half or a sibling `SkillPanelEntry` component; implementer picks the smaller surface) publishing a panel-origin payload to `SkillDragController`.
- Refresh on Show from the selected character's class (same `ClassesRepo.Get` pattern as `MainHudPanel.Refresh`).
- **Drag-safety contract**: `SkillsPanel` must NOT be registered with the polled `UIRefreshDriver` in a way that rebuilds/destroys its rows while open — a destroyed row mid-drag kills the drag source. Either exclude it from polling entirely (its content only depends on the class, which cannot change while it is open — refresh on `Show()` suffices) or make `Refresh()` a no-op while the class is unchanged or a drag is active. Prefer the former: rebuild only in `Show()`.

**Current state**: fields `skillButtons`/`skillCooldownFills`/`skillHotkeyLabels`/`skillLabels` (lines 39–42), binding in `BindCombatControls` (67–85), digit-1/2/3 hotkeys in `Update` (97–99), `TriggerSkillSlot` (193), `RefreshSkillSlots` (203), `RefreshSkillCooldowns` (218), `SkillAtSlot` (243).

**Modifications**:
- Remove all of the above (hotkeys and trigger/cooldown logic move to `SkillBarPanel`; `SkillAtSlot`'s priority-sort logic is reused by the SaveManager migration instead).

### 10. UIBuilder — build the bar, the panel, and drop the old slots

**File**: `Assets/Scripts/UI/UIBuilder.cs`

## Technical Considerations

- **Pattern Usage**: `MainHudPanel` conventions for the new panels (`[HideInInspector]` baked references, `Refresh()` pulling from `GameManager`); `BuildSubPanel<T>` for SkillsPanel; existing migration ladder in `SaveManager`; existing mutation-then-save pattern in `GameManager`.
- **Layer Placement**: `SkillBar` field + slot-count constant → Data (engine-free — `List<string>` only). Validation → Core (`StateInvariantValidator`, engine-free). Mutation API + migration → Managers. All new components → `IdleCloud.UI`. No new asmdef references needed.
- **Determinism**: none affected — no randomness, no offline-rate impact (bar is manual-trigger UI; auto-combat and `EfficiencySnapshot` unchanged).
- **Editor Authoring Impact**: after code changes the user must rebake (`IdleCloud > UI > Bake UI Prefab`) **with `PersistentGame.unity` as the active scene** (note: v0.2.0 known deviation — the current baked GameUI instance sits in `Maps/FirstMap.unity`; this bake is the natural moment to land the instance in PersistentGame, but moving it is the user's call — NOTED (not done), see `docs/STATE.md` open item). Then Play-mode check: drag from Skills panel → slot, slot → slot swap, drag-off clear, hotkeys 1–8, cooldown fills, click-triggers still reach combat.
- **Save Compatibility**: `SaveSchemaVersion` 2 → 3. Migration seeds `SkillBar` from priority-sorted class skills (slots 0–2) — old saves keep today's exact layout; an unconditional normalization pass repairs malformed bars on every load. Pre-migration backup fires only in Editor/development builds (`#if` guard, SaveManager.cs:219) — fine pre-release. `SaveCompatibilityTests` must gain a v2-fixture → v3 assertion.
- **Edge Cases**:
  - Class has fewer skills (currently 3) than slots (8) → trailing slots empty; empty slot click/hotkey = no-op.
  - Save references a skill id no longer in the class (content drift) → the unconditional normalization pass clears the entry to `null` (validator only reports).
  - Assigning a skill already on the bar → moves it (old slot nulled), never duplicates.
  - Drag ends over nothing: from panel → no-op; from bar slot → clear.
  - Character switch → bar re-renders from the newly selected character's `SkillBar`.
  - No active combat → trigger command still enqueues exactly as today (no behavior change).
  - Buff/zero-damage skills: today's `TriggerSkillSlot` refuses `DamageMultiplier <= 0` — carry that gate over unchanged (NOTED: it makes the Wizard-style Buff skill untriggerable manually; pre-existing behavior, not this plan's problem).

## Type Definitions

- `Character.SkillBar : List<string>` (Data) — length 8, entries `ClassSkillDef.Id` or `null`.
- Slot-count constant in Data (single source of truth; UI layout mirrors it).
- Drag payload (UI-internal, on `SkillDragController`): skill id + origin (panel entry | bar slot index).

## Backward Compatibility

Old saves (schema ≤ 2) load through the new `< 3` migration step: bar seeded to reproduce the current auto-derived 3-button layout. The unconditional normalization pass additionally repairs any malformed bar (wrong length, stale ids, duplicates) on every load regardless of schema version. Backup file is written automatically in Editor/development builds only. No field removed or renamed, so forward serialization is additive — Newtonsoft ignores the new field when absent and writes it thereafter.

- `Assets/Scripts/Tests/EditMode/StateAndValidationTests.cs`: SkillBar invariants (length, membership, no duplicates); `CreateCharacter` produces a seeded, valid bar.
