# Drops & Loot Pickup (Ground Loot Flow) Implementation Plan

**Version:** 0.8.0 · **Trello cards:** "Basic drop tables", "Loot pickup / claim flow", "Item quantity range" (Drops), touches "Combat feedback"

## Overview

Killing a mob currently rolls its drop table and silently teleports the items into the bank/inventory — the player never sees loot. This feature makes loot physical: killed mobs drop a **generic loot bag** on their tile; the player **clicks the bag** to walk over and pick it up (or, with AutoLoot on, it is **auto-vacuumed after a short delay**). Pickup always lands in the **character inventory**; what doesn't fit **stays in the bag** until a despawn timer expires. Pickup feedback comes as world-space popup text plus a HUD loot feed. The slime's table is reworked to drop `slime_goo`, closing the kill → loot → craft loop (Leather armor recipes) on FirstMap. The `DropTable` format additionally gains a **tertiary drop list** (structure only, no content yet).

## Problem Statement

- `ActiveCombatCoordinator.ApplyReward` (Assets/Scripts/Managers/Gameplay/ActiveCombatCoordinator.cs:140) deposits loot instantly and invisibly; `CombatView.cs:222` uses the reward only to detect the kill.
- Overflow (full inventory/bank) is silently destroyed during active play; only the offline report surfaces it.
- The slime drops `wolf_pelt`, which feeds no reachable recipe; `slime_goo` recipes (Leather Cap/Tunic/Pants, level 1–3) have no source.
- The Trello "Tertiary drops" card would force a second table-format migration later if the structure isn't added now.

## User Decisions (locked during discovery)

1. Ground drops with auto-pickup; **AutoLoot toggle = auto-vacuum** (no longer "route to bank").
2. Overflow stays in the bag on the ground until despawn.
3. Generic loot bag sprite (contents not visible from a distance).
4. **Items only** drop on the ground; coins → bank and XP → character stay instant.
5. Pickup destination is **always the character inventory** (bank stays a deposit action).
6. Lifetime: tunable despawn timer; unclaimed loot is **lost on map travel/quit** — no save-format change.
7. Manual pickup by **clicking the bag** (player paths to it, like clicking a mob).
8. Feedback: **pickup popup text** (world-space) + **HUD loot feed** (also carries "inventory full" warnings).
9. Tertiary drops: **structure only** (types, rolls, authoring, validation) — no tertiary content.
10. Gathering (mining/woodcutting) keeps its instant flow.
11. Slime table reworked to **`slime_goo` main drops** (+ occasional `small_hp_potion`); `wolf_pelt` removed (belongs on a future wolf).
12. Drop tables authored as **ScriptableObject assets** (`MonsterDefinitionAsset` + `DropTableDefinitionAsset` in `ContentRegistryAsset`), overriding the code repos.
13. Offline loot aligns with the new rule: **inventory-first**, overflow shown in the offline report.

## Solution Architecture

The loot lifecycle splits into three stages across the existing layers (downward flow preserved):

1. **Roll (Core, unchanged mechanics):** `DropSystem.RollDropTable` keeps rolling per kill inside `ActiveCombatCoordinator.Tick`. New: a tertiary pass (independent fixed-chance rolls per kill) after the main weighted pass.
2. **Spawn & hold (Managers, new):** `ActiveCombatCoordinator.ApplyReward` stops depositing item loot (coins/XP unchanged). Rolled stacks flow into a new plain-C# **`LootDropManager`** (Managers/Gameplay) that owns runtime-only ground-loot records — `{dropId, mapId, tile/world position, stacks, spawnedAtMs}` — with timestamp-based despawn and auto-vacuum scheduling (no MonoBehaviour, EditMode-testable). It exposes C# events (`LootSpawned`, `LootPickedUp`, `LootExpired`, `LootOverflowed`) for View/UI.
3. **Render & pick up (View/UI):** A new **`LootBagView`** component (spawned from a `[SerializeField]` prefab reference) renders the bag on the mob's tile; `PlayerController` learns bag click-targets exactly like `FindCombatTargetAt`; arrival triggers `GameManager.TryPickupLoot(dropId)`, which runs the `LootDropManager` transaction (`Inventory.AddToInventory` per stack — fitting stacks leave the bag, the remainder stays) and commits the account. HUD `LootFeedPanel` and a world-space pickup popup subscribe to the events.

Ground loot is deliberately **not** persisted: it lives only in `LootDropManager`, is cleared on map travel/quit, and never touches the save format.

## Implementation Details

### 1. `DropTable` tertiary structure

**File**: `Assets/Scripts/Data/State/GameTypes.cs` (~line 503)

**Current state**: `DropTable { Always: List<DropItem>, Main: WeightedTable }`.

**Modifications**:
- Add `Tertiary: List<DropEntry>` (reuses the existing chance-based `DropEntry` shape: ItemId, Chance, Min, Max). Empty/null = no tertiary drops.

### 2. DropSystem tertiary rolls

**File**: `Assets/Scripts/Core/DropSystem.cs`

**Modifications**:
- `RollDropTable` (line 55): after the main weighted pass, roll each tertiary entry independently once per kill (`rng.NextDouble() < Chance * (1 + dropBonus)`-style; exact bonus interaction mirrors how `RollDrops` applies `dropMult` — clamp to 1.0). Quantity via the existing `UniformInt`.
- `ExpectedDropTable` (line 86): add the expected-value contribution `kills * effectiveChance * Avg(Min, Max)` per tertiary entry, preserving the active/offline parity contract.
- `ValidateDropTable` (line 150): validate tertiary entries (item ID present, chance in [0,1], valid min/max range) — mirror `ValidateDropEntry`.

### 3. Offline loot goes inventory-first

**File**: `Assets/Scripts/Core/Offline.cs` (~lines 62–71)

**Current state**: per-character loot deposits to the shared bank; overflow → `report.LootOverflow`.

**Modifications**:
- Deposit each character's loot into **that character's inventory** (`Inventory.AddToInventory`); overflow → `report.LootOverflow` unchanged. Coins keep going to the bank. The updated character (already cloned for `Activity` reset) carries the new inventory.
- `OfflineReportPanel.cs` (~line 90) currently labels overflow "BANK FULL" — reword to inventory-full to match the new destination.

### 4. ActiveCombatCoordinator: stop instant loot

**File**: `Assets/Scripts/Managers/Gameplay/ActiveCombatCoordinator.cs`

**Current state**: `ApplyReward` (line 140) routes loot AutoLoot→bank / else→inventory; overflow destroyed.

**Modifications**:
- `ApplyReward` applies only XP and coins. Item stacks are no longer deposited; `CombatReward.Overflow` is no longer filled here (overflow becomes a pickup-time concept).
- **Per-actor loot attribution**: today `CountEnemyKilledEvents` (line 83) discards each `EnemyKilled` event's `TargetId` and `MergeReward` aggregates everything. Instead, roll and return loot **per defeated actor**: `CombatReward` gains `List<KillLootRecord> KillLoot` where `KillLootRecord = {ActorEntityId, MonsterId, List<ItemStack> Stacks}` — one record per `EnemyKilled` event, built from that event's `TargetId`. XP/coins stay aggregated as today. The caller spawns one bag per record at that actor's tile.
- The `_items` unknown-item guard stays (validated content makes it near-dead, but keep the check and surface unknowns via the loot feed rather than silently).

### 5. LootDropManager (new)

**File**: `Assets/Scripts/Managers/Gameplay/LootDropManager.cs` (new)

- Plain C# class, same style as the coordinators: constructed with the item-def dictionary (from `RuntimeContent`), timestamp-driven, no UnityEngine types. **Owned by `GameManager`** (like the coordinators), not by the View.
- API surface (names indicative): `Spawn(mapId, tile+floor, List<ItemStack>, nowMs)` → dropId; `TryPickup(Account, characterId, dropId)` → `LootPickupResult` (updated `Account`, picked stacks, remaining stacks) — transaction mirrors `Crafting`'s exactness: per-stack `Inventory.AddToInventory`, partial success allowed, remainder stays in the record; `CollectDue(nowMs, autoLootEnabled)` → returns dropIds due for auto-vacuum and removes records past their lifetime (firing `LootExpired`); `ClearMap(mapId)` for travel/quit.
- **Authoritative commit path lives in `GameManager`** (the class that owns the private `CommitAccount`, cf. `ResolveCombatKill` at GameManager.cs:626): public API surface — `SpawnLoot(mapId, tile+floor, stacks)` → dropId (called by CombatView per `KillLootRecord`), `TryPickupLoot(dropId)` (manual, called by View on arrival), and the auto-vacuum drain (called from GameManager's own per-frame/heartbeat update, **independent of combat ticking**). Pickup paths run `LootDropManager.TryPickup` and commit via `CommitAccount` with a new `SessionCommand`/`SessionEventKind` (follow the `ResolveCombatKillCommand` example). GameManager **publicly relays the loot lifecycle events** (`LootSpawned`, `LootPickedUp`, `LootExpired`, plus `LootCleared(mapId)` on map clear) so View/UI subscribe to GameManager only — they never touch the service directly.
- Tunables (`DespawnMs`, `AutoVacuumDelayMs`) live in a `LootDropConfig` set from `[SerializeField]` Inspector fields on the GameManager host — no hard-coded gameplay values.
- Events (all carry `dropId`): `LootSpawned(dropId, record)`, `LootPickedUp(dropId, characterId, pickedStacks, remainingStacks)`, `LootExpired(dropId, lostStacks)` — remaining-stacks lets the View keep or destroy the bag; an empty-after-pickup bag is removed.

### 7. LootBagView (new)

**File**: `Assets/Scripts/View/LootBagView.cs` (new)

- Component on a hand-authored prefab `Assets/Prefabs/World/LootBag.prefab` (user creates it in the Editor; `[SerializeField]` sprite placeholder, swap art later — no programmatic object generation beyond instantiating the prefab).
- Carries: click trigger collider (same pattern as `CombatTargetView`'s auto-created `CircleCollider2D`, clickRadius/offset serialized), the owning `dropId`, an exposed `LogicalPosition` (ground cell XY + floor/height, set at spawn), and `public UnityEvent OnPickedUp` / `OnSpawned` for sound/VFX wiring in the Inspector.
- **Sorting bridge** (`IdleCloud.View` cannot reference Assembly-CSharp, and `WorldObjectSortController.floorIndex` is Inspector-only): a new `LootBagSortController` in `Assets/Iso/Sorting/` (Assembly-CSharp) that mirrors `EnemySortController` exactly — sits on the prefab beside a `SortingGroup`, reads `LootBagView.LogicalPosition` every frame, and feeds the existing protected sort calculator. **No change to the sort formula or settings.** `GroundLootRecord` therefore stores tile + floor, and the spawn call passes the defeated actor's floor.
- Optional despawn warning (e.g. blink in the last seconds) driven by a serialized threshold — visual only.

### 8. CombatView / PlayerController integration

**Files**: `Assets/Scripts/View/CombatView.cs`, `Assets/Scripts/View/PlayerController.cs`

**Modifications**:
- **Loot time never depends on combat**: `CombatView.Update` early-returns whenever no target exists (`CombatView.cs:113`), so despawn/vacuum must NOT be ticked from the combat tick. GameManager's own update/heartbeat path advances `LootDropManager` time and drains auto-vacuum (§5) — bags keep counting down after combat ends.
- `CombatView`: on a reward, calls `GameManager.SpawnLoot` **once per `KillLootRecord`** at that defeated actor's logical tile+floor (actor looked up in `_targets` by `ActorEntityId`; the current-target defeat branch at `CombatView.cs:222` is only one such case — AoE multi-kills produce several records). Instantiates the `[SerializeField] LootBagView lootBagPrefab` per returned dropId; destroys/updates bag instances on the relayed `LootPickedUp`/`LootExpired`/`LootCleared` events (by dropId, using remaining-stacks to decide).
- `PlayerController`: add bag click detection mirroring `FindCombatTargetAt` (line 190) — clicking a bag raises `LootTargetSelected(LootBagView)` (same shape as `CombatTargetSelected`, line 62) and paths to its cell. **Arrival signaling**: `PlayerController` exposes a destination-reached notification for the loot intent (completion or cancellation — a new click, combat engagement, or an unreachable path clears the intent); on completion the listening View calls `GameManager.TryPickupLoot(dropId)`. The plan explicitly does NOT commit accounts from the View — GameManager owns the commit (§5).
- **Map lifecycle**: the travel/map-change path in GameManager (and player quit) calls `LootDropManager.ClearMap` so bags never leak across maps; the View clears its bag instances on the same signal.
- Pickup popup: reuse the existing world-space feedback pattern (`CombatFeedbackView`) to show "+N Item Name" above the player on `LootPickedUp`; item display names resolved from `RuntimeContent.Items`.

## Technical Considerations

- **Pattern Usage**: `LootDropManager` copies the coordinator style (plain class, timestamp ticks, account-in/account-out transactions); `LootBagView` copies `CombatTargetView` (click collider, serialized geometry) and `WorldObjectSortController` usage; `LootFeedPanel` copies the existing UIBuilder panel pattern; content override copies the existing `ContentRegistryAsset.Apply` path.
- **Layer Placement**: types → Data (`GameTypes`); rolls/offline → Core (engine-free, no UnityEngine); loot registry/transactions → Managers; bag + input → View; feed → UI. Downward-only preserved: Core never knows bags exist.
- **Determinism**: all randomness stays in `DropSystem` behind `IRandomSource`. Despawn/auto-vacuum use a **GameManager-owned monotonic timestamp** (the same Unix-ms base its heartbeat/commit path already uses) — not CombatView's `_simTimeMs` and not frame counts. Active/offline parity: tertiary EV math mirrors the stochastic path exactly (same effective chance formula).
- **Editor Authoring Impact** (user steps in live Editor): create `LootBag.prefab` with placeholder sprite; create + wire `DropTable_Slime.asset`, `Monster_Slime.asset` into `ContentRegistryAsset`; UI rebake for the loot feed; assign `lootBagPrefab` on the CombatView host and the `LootDropConfig` tuning fields (`DespawnMs`, `AutoVacuumDelayMs`) on the **GameManager host**; Play-mode verify.
- **Save Compatibility**: none — ground loot is runtime-only; `Account.AutoLoot` keeps its serialized shape (meaning changes, value doesn't). `SaveCompatibilityTests` unaffected.
- **Edge Cases**: kill while inventory full (bag holds everything until space frees, then partial pickup); multiple kills on one tile (either merge into one bag or offset — recommend merge into the nearest existing bag record); player leaves map with bags present (`ClearMap`); AutoLoot toggled while bags exist (vacuum applies from next tick); target defeated by AoE with several simultaneous kills in one tick (one bag per defeated actor's tile); despawn during path-to-bag (arrival finds no record → no-op + feed notice).
