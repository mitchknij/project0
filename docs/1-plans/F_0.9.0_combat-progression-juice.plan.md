# Combat & Progression Juice Implementation Plan (v0.9.0)

## Overview

Adds moment-to-moment feedback ("juice") to combat, progression, and gathering: a gold level-up banner in the HUD, "+N coins" popups on kills, visually distinct critical hits, an XP-bar pulse on every XP gain, a vacuum fly-to-player animation for auto-looted bags, a spawn bounce for landing loot bags, and shake/crumble feedback on gathering nodes. All changes are presentation-side plus a thin Managers-layer plumbing pass that stops discarding level-up information the Core already computes.

User decisions (discovery round): level-up moment is a **HUD banner only** (no world popup); XP feedback is an **XP-bar pulse** (no "+8 XP" popups); feedback fires for **kills, gathering, and crafting** wherever XP/coins actually flow; node feedback is **hit-shake on every swing + crumble-puff per successful gather** — depletion/respawn visuals are deferred because no depletion mechanic exists in Core.

## Problem Statement

- `CharacterHelper.ApplyCharacterXp` / `ApplySkillXp` return `PreviousLevel`/`NewLevel`/`LevelsGained`, but both coordinators call `.Character` and discard the result (`ActiveCombatCoordinator.cs:174-175`, `ActiveGatheringCoordinator.cs:49`) — level-ups are completely invisible in-game.
- Coins go silently to the bank; `CombatReward.Coins` is aggregated per tick but never surfaced per kill.
- Crits already carry `CombatEvent.Critical` and get a bigger gold popup (`CombatFeedbackView.SpawnPopup`), but read as barely different from normal hits.
- Auto-looted bags vanish instantly (`CombatView.DestroyLootBag`); bag spawns appear without any animation; gathering nodes give no per-swing physical feedback.

## Solution Architecture

Three thin layers, strictly downward:

1. **Managers plumbing** — capture what Core already computes: per-kill coins on `KillLootRecord`, character/skill level transitions on the two tick results, and two new normalized GameManager events (`XpAwarded`, `LevelUp`) as the single subscription point for UI. Add a `Vacuum` flag to `LootPickedUpEvent` so the View can distinguish vacuum pickups from manual ones.
2. **View juice** — extend the existing placeholder-VFX patterns in `CombatFeedbackView` (coin popups, crit emphasis), `LootBagView`/`CombatView` (spawn bounce, vacuum tween), and `GatheringNodeView`/`GatheringView` (shake + crumble-puff). No new prefabs, no external assets; reuse the runtime `PixelSprite` + coroutine style already in place.
3. **UI** — new `LevelUpBannerPanel` built by `UIBuilder` (rebake required), and an XP-bar pulse in `MainHudPanel`.

No Core or Data changes. No save-shape changes. No new randomness.

## Implementation Details

### 1. ActiveCombatCoordinator — per-kill coins + level transitions

**File**: `Assets/Scripts/Managers/Gameplay/ActiveCombatCoordinator.cs`

**Current state**: `CombatReward` holds aggregate `CharacterXp`/`CombatSkillXp`/`Coins`; `KillLootRecord` holds only item stacks; `ApplyReward` (line 171) discards both `CharacterXpResult` and `SkillXpResult`.

**Modifications**:
- Add `Coins` field to `KillLootRecord`; populate it in `Tick` (around line 103) from the already-computed `killReward.Coins` — **no extra `_random` calls** (RNG call order must not change; determinism).
- Add character-level and combat-skill-level transition fields to `CombatReward` (previous/new level for both). Populate in `ApplyReward` from the `CharacterXpResult`/`SkillXpResult` return values instead of discarding them.

### 2. ActiveGatheringCoordinator — skill level transitions

**File**: `Assets/Scripts/Managers/Gameplay/ActiveGatheringCoordinator.cs`

**Current state**: `ActiveGatheringTickResult` has `Account`/`Simulation`/`Overflow`; line 49 discards the `SkillXpResult`.

**Modifications**:
- Add skill level-up fields to `ActiveGatheringTickResult` (the `SkillId` plus previous/new level), populated from the `ApplySkillXp` result in `Tick`.

### 3. GameManager — normalized progression events + vacuum flag

**File**: `Assets/Scripts/Managers/GameManager.cs`

**Current state**: publishes `ActiveCombatResolved`, `ActiveGatheringResolved`, loot events, `CraftCompleted` (lines 148-155). Vacuum pickup path and manual `TryPickupLoot` both go through `LootDropManager` (around lines 620-644).

**Modifications**:
- Two new public events with plain-C# payload classes (pattern: the event classes in `LootDropManager.cs`): `XpAwarded` (character XP, skill XP, skill id) and `LevelUp` (kind character-or-skill, skill id, previous level, new level — one event per level actually crossed, so a multi-level jump queues multiple banners).
- Fire them from `TickActiveCombat` (when `result.Reward` is present, around line 720) and `TickActiveGathering` (when `SuccessfulActions > 0`, around line 760), reading the new coordinator fields. Offline rewards deliberately do **not** fire these — `OfflineReportPanel` already reports offline level-ups.
- Pass a vacuum/manual flag down the pickup call path so `LootPickedUpEvent` carries it (see §4).

### 4. LootDropManager — vacuum flag on pickup event

**File**: `Assets/Scripts/Managers/Gameplay/LootDropManager.cs`

**Modifications**:
- Add `Vacuum` bool to `LootPickedUpEvent`; the pickup entry point gains a parameter (or overload) so GameManager's vacuum path sets it true and manual pickup sets it false. Reference-sweep all constructors/consumers of `LootPickedUpEvent`.

### 5. CombatFeedbackView — coin popups + crit emphasis

**File**: `Assets/Scripts/View/CombatFeedbackView.cs`

**Current state**: subscribes to `ActiveCombatResolved` (line 75); `SpawnPopup` (line 418) already branches on `critical` (font 4.8 vs 3.6, gold vs red, "!" suffix); `AnimatePopup` only rises and fades.

**Modifications**:
- **Coins (punt 2)**: in `HandleCombatResult`, when `result.Reward?.KillLoot` is present, spawn a gold "+N coins" floating text per `KillLootRecord` with `Coins > 0`, positioned at the killed actor (`_targets` lookup by `ActorEntityId`, player-position fallback). Reuse `SpawnFloatingText`; add `[SerializeField]` coin color/font size.
- **Crit (punt 3)**: rework `SpawnPopup`/`AnimatePopup` so crits get an orange palette, a larger size, and a scale-punch (pop-in with overshoot, then settle) versus the plain rise-and-fade of normal hits. All tunables (`critColor`, crit font size, punch scale/duration) as `[SerializeField]` fields.

### 6. LootBagView — spawn bounce + pickup tween

**File**: `Assets/Scripts/View/LootBagView.cs`

**Current state**: `ConfigureRuntimeDrop` shows the bag instantly; `OnSpawned`/`OnPickedUp` UnityEvents already exist.

**Modifications**:
- **Bounce (punt 7)**: on `ConfigureRuntimeDrop`, run a scale 0→1 overshoot coroutine (`[SerializeField]` duration + overshoot factor). Scale the renderer child(ren), not physics-critical state — colliders stay live from frame one.
- **Vacuum tween (punt 6)**: new public method that flies the bag to a world position (ease-in, slight shrink) and destroys/notifies on arrival. Colliders are already disabled by `NotifyPickedUp` before the tween starts, so a mid-flight bag can't be clicked.

### 7. CombatView — route pickup through the tween

**File**: `Assets/Scripts/View/CombatView.cs`

**Current state**: `HandleLootPickedUp` (line 380) → `NotifyPickedUp` → instant `DestroyLootBag`.

**Modifications**:
- When the pickup event has `Vacuum == true`, the bag exists **and `RemainingStacks` is empty**: trigger the bag's fly-to-player tween (target = the player reference CombatView already holds for loot targeting) and remove it from `_lootBags` immediately (the record is dead; the flying bag is pure cosmetics). Partial pickups (`RemainingStacks` non-empty — e.g. inventory full or unknown item ids) keep today's behavior exactly: `NotifyPickedUp(true)`, bag stays visible and clickable, no tween. Manual pickups keep the current instant destroy (player is standing on the bag; a tween would be invisible). `LootExpired`/`LootCleared` paths unchanged.
- Sorting note: during the short flight the bag's `LootBagSortController` keeps the origin-tile sort order — acceptable for a sub-half-second cosmetic flight; do **not** touch the sorting system (protected, ARCHI §9).

### 8. GatheringNodeView — hit shake + crumble-puff

**File**: `Assets/Scripts/View/GatheringNodeView.cs`

**Current state**: owns the runtime progress bar; `_renderer` is the sprite (possibly on an offset "Visual" child).

**Modifications**:
- New public methods: a hit-shake (brief positional wiggle of the **sprite child's local position** — never the root transform, which `WorldObjectSortController` snaps to cell centers) and a crumble-puff (a few short-lived `PixelSprite` fragments in the node's palette that scatter/fall and fade, same placeholder-VFX idiom as `CombatFeedbackView`). Tunables (`shakeDuration`, `shakeAmplitude`, puff count/color) as `[SerializeField]` fields.
- Coroutines must restore the exact original local position when interrupted/re-triggered (rapid swings).

### 9. GatheringView — route gathering events to node effects

**File**: `Assets/Scripts/View/GatheringView.cs`

**Current state**: there is no event-subscription handler — `Update()` calls `manager.TickActiveGathering(...)` synchronously (line 81) and already iterates `result.Simulation.Events` in a loop directly below it (lines 85-97, currently handling `MovementRequested`/`CommandRejected`).

**Modifications**:
- Extend that existing event loop in `Update()`: every `AttemptResolved` triggers the node's hit-shake (hit **and** miss — the swing lands on the rock either way, per the user-approved behavior); every `ResourceGathered` triggers the crumble-puff. This runs immediately after `TickActiveGathering`, before the early returns — which is safe because these event kinds only exist on genuinely resolved ticks. Cap effect triggers per tick (catch-up batches after tab-out must not fire 200 shakes — same rationale as the existing "one Miss popup per tick" rule in `CombatFeedbackView`).

### 10. LevelUpBannerPanel — gold HUD banner (new)

**File**: `Assets/Scripts/UI/LevelUpBannerPanel.cs` (new)

- Plain component per the panel pattern: TMP gold text ("LEVEL 6!" / "MINING LEVEL 4!"), CanvasGroup, screen-center anchored, non-interactive (no raycast blocking).
- **The root GameObject stays active at all times** (an inactive object never runs `Update`, so the lazy subscription would never happen). Idle-invisibility is done via `CanvasGroup.alpha = 0` with `interactable`/`blocksRaycasts` false — never `SetActive(false)`.
- Subscribes to `GameManager.LevelUp` with the lazy `GameManager.Instance` poll pattern used by `CombatFeedbackView`/`LootFeedPanel`; unsubscribes in `OnDestroy`.
- Animation: pop-in scale + brief gold flash, hold, fade out (coroutine; `[SerializeField]` durations). Multiple level-ups queue and play sequentially.
- Skill display names resolved from the skill id (fallback: the enum name).

### 11. UIBuilder + UITheme — build & theme the banner

**Files**: `Assets/Scripts/UI/UIBuilder.cs`, `Assets/Scripts/UI/UITheme.cs`

- `UIBuilder` constructs the banner (active GameObject, invisible via CanvasGroup alpha 0 — see §10; rendered above panels, blocking nothing interactive) and wires it into the baked hierarchy; layout constants land in `UITheme.Layout`, colors from the existing palette (`TextGold`).
- **Rebake required**: `IdleCloud > UI > Bake UI Prefab` with `PersistentGame.unity` as active scene (user step).

### 12. MainHudPanel — XP-bar pulse

**File**: `Assets/Scripts/UI/MainHudPanel.cs`

**Current state**: `xpFill`/`xpLabel` refreshed by `Refresh()`; no event subscriptions.

**Modifications**:
- Subscribe to `GameManager.XpAwarded` (same lazy pattern); on any XP (character or skill — per user decision the pulse fires for gathering skill XP too), run a short pulse coroutine on the XP bar (brightness flash and/or slight vertical scale punch), tunables serialized. Coroutine restores exact base color/scale; re-trigger restarts cleanly.

## Technical Considerations

- **Pattern Usage**: placeholder-VFX idiom (runtime `PixelSprite`, coroutines, serialized tunables) from `CombatFeedbackView`; event-relay pattern from `LootDropManager`/`GameManager`; lazy `GameManager.Instance` subscription from `LootFeedPanel`/`CombatFeedbackView`; panel construction via `UIBuilder` + `UITheme.Layout`.
- **Layer Placement**: §1-4 in `IdleCloud.Managers` (plain C#, no engine types in the new payload classes); §5-9 in `IdleCloud.View`; §10-12 in `IdleCloud.UI`. Zero Core/Data changes; downward flow preserved (UI reads Managers events; nothing references UI/View).
- **Determinism**: no new `IRandomSource` draws and no reordering of existing draws — `KillLootRecord.Coins` reuses the already-rolled value. Crumble-puff scatter may use `UnityEngine.Random` freely (View-side cosmetics, outside the deterministic sim). Offline math untouched.
- **Editor Authoring Impact**: all tunables `[SerializeField]`; no `GameObject.Find`. User steps: UI rebake (banner + HUD pulse fields) with PersistentGame active, save, Play-mode verify. `LootBagView`/`GatheringNodeView` prefab instances pick up new serialized defaults automatically — re-check hand-tuned prefab overrides are untouched. If the Unity Editor is closed when new file §10 is added, its `.meta`/csproj entry may need the known manual-glob workaround (ARCHI §6).
- **Save Compatibility**: none — all new state is runtime-only (tick results, events, view animation). No snapshot/save shape change; `SaveCompatibilityTests` unaffected.
- **Edge Cases**:
  - Multi-level jump in one XP award → one `LevelUp` event per level, banner queue plays them in order.
  - Multiple kills in one tick → one coin popup per `KillLootRecord`; positions differ per killed actor.
  - Gathering catch-up batch after tab-out → shake/crumble capped per tick.
  - Bag despawns or map travel while a vacuum tween is in flight → tween coroutine dies with the destroyed object; `LootCleared`/`LootExpired` destroy paths must also cancel cleanly.
  - Partial vacuum pickup (inventory full / unknown item id) → bag must **stay** on the ground, visible and clickable (`RemainingStacks` guard in §7); only a fully-emptied bag flies away.
  - Crafting currently grants no XP → `XpAwarded` simply never fires from crafts; craft item popup + feed already exist. No dead code written for it.
  - Player dead/respawning during a pending banner → banner is HUD-level and independent of world state; no coupling.

## Type Definitions

- `XpAwardedEvent` (Managers, plain C#): character XP amount, skill XP amount, skill id.
- `LevelUpEvent` (Managers, plain C#): kind (character | skill), skill id (null for character), previous level, new level.
- Field additions (no new types): `KillLootRecord.Coins`; level-transition fields on `CombatReward` and `ActiveGatheringTickResult`; `Vacuum` on `LootPickedUpEvent`.

## Performance & Cost Impact

Negligible: a handful of short-lived GameObjects/coroutines per kill/gather, same order of magnitude as existing popups. Per-tick caps prevent catch-up storms. No per-frame allocations outside animation lifetimes.

## Backward Compatibility

No save impact. All contract changes are additive fields on Managers-layer runtime types; existing consumers compile unchanged.
