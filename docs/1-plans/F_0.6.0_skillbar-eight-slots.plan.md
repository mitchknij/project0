# Skillbar Eight Slots Implementation Plan

## Overview

Widen the manual skillbar from 4 to 8 slots — the count the v0.4.0 plan specified but which shipped as 4 — and extend hotkeys to digits 1–8. The auto-combat priority rule is **unchanged**: auto-combat keeps scanning the bar left→right and casting the first eligible skill, so slot 0 stays highest priority. On top of that, each slot gains a very subtle highlight that fades out quickly whenever auto-combat selects that slot, making the existing priority rule visible without adding any new state or new selection semantics.

## Problem Statement

- `Character.SkillBarSlots = 4` (`GameTypes.cs:226`) caps the bar at 4, while `SkillBarPanel`'s own doc-comment (`SkillBarPanel.cs:11`) still claims "eight-slot" — code and intent disagree.
- `AutoCombatPolicy.NextAutoSkill` scans `Math.Min(4, ...)` with a **hardcoded literal** (`AutoCombatPolicy.cs:40`) instead of the shared constant. Widening the constant alone would leave auto-combat blind to slots 5–8 — a silent correctness bug.
- Hotkeys stop at digit 4 (`SkillBarPanel.cs:64-67`).
- The priority rule (leftmost eligible skill wins) is invisible in play — the player gets no feedback about which slot auto-combat is actually firing.

## Solution Architecture

Three of the four consumers are already constant-driven and need **no changes**: `UIBuilder.BuildSkillBar` sizes and populates from `UITheme.Layout.SkillBarSlotCount` (`UIBuilder.cs:498-515`), `UITheme.Layout.SkillBarSlotCount` mirrors the Data constant (`UITheme.cs:45`), and `SaveManager.NormalizeSkillBars` rebuilds every bar to exactly `Character.SkillBarSlots` entries on every load (`SaveManager.cs:242-253`). So the widening is: **one constant, one literal, four hotkeys.**

The highlight reads `state.SkillRuntime.LastAutoSelection` (an `AutoSkillSelectionDiagnostics` carrying `SelectedSlotIndex`), which `ActiveSim.cs:108` writes inside the auto-rotation branch, via the `GameManager.ActiveCombatState` path `SkillBarPanel` already uses.

**`SelectedSlotIndex` alone cannot drive the pulse.** It is retained between auto decisions and holds the *same* value whenever auto-combat re-picks the same slot — the normal case, since classes have ~4 skills and slot 0 wins most ticks. Detecting "the selected slot changed" would therefore pulse once and then stay silent, including across encounters. The highlight instead keys on a **per-selection sequence id** stamped on each successful auto cast, so a repeated slot still produces a fresh pulse. This costs one `long` field in Data and two lines in Core.

Save compatibility needs no schema bump: the unconditional normalization pass already pads short bars with `null`, so a 4-entry v3 save becomes an 8-entry bar with the existing 4 assignments intact.

### 3. Extend hotkeys to 1–8 and add the auto-highlight

**File**: `Assets/Scripts/UI/SkillBarPanel.cs` (275 lines — Read in full before editing, per C1)

**Current state**: `Update()` (lines 58-71) maps `digit1Key`–`digit4Key` to `TriggerSkillSlot(0..3)`, then calls `RefreshSkillCooldowns()` and `RefreshFeedback()`. A per-slot transient feedback channel already exists — `_feedbackUntil` / `_feedbackText` / `_feedbackColor` float+string+Color arrays (lines 22-24), sized in `BindCombatEvents` (lines 78-81), written by `ShowFeedback` (line 162), and consumed in `RefreshFeedback` (line 171) which overwrites `slotNameLabels[i].text`.

**Modifications**:
- **Hotkeys**: extend the digit block (lines 64-67) with `digit5Key`–`digit8Key` → `TriggerSkillSlot(4..7)`. Prefer driving this from a static `Key[]` table indexed by slot over eight more hand-written lines, so the block can never drift from the constant again.
- **Highlight state**: add a `[HideInInspector] public Image[] slotAutoHighlights;` reference array (populated by `UIBuilder`, §4) and a `private float[] _autoHighlightUntil;` timer array, sized alongside the existing feedback arrays in `BindCombatEvents` (line 78-81).
- **Detect selection**: in `Update`, read `GameManager.Instance?.ActiveCombatState?.SkillRuntime?.LastAutoSelection`. Stamp `_autoHighlightUntil[SelectedSlotIndex] = Time.unscaledTime + UITheme.Layout.SkillBarAutoHighlightSeconds` when `SelectedSlotIndex >= 0` **and `SelectionSequenceId` differs from the last observed value** (track `private long _lastAutoSelectionSequenceId = 0;`). Keying on the sequence id — not on the slot index — is what makes a repeated same-slot cast pulse again (see §6). Re-stamping unconditionally every frame would make the highlight solid rather than fading.
- **Render the fade**: a new `RefreshAutoHighlight()` called from `Update` sets each overlay Image's alpha to `Mathf.Clamp01((_autoHighlightUntil[i] - Time.unscaledTime) / duration)` scaled by a peak alpha constant, so it decays linearly to invisible. Deliberately a **separate visual channel from `_feedback*`** — the feedback channel hijacks the name label text, whereas the highlight must not disturb the skill name the player is reading.
- Do **not** write to `slotButtons[i].image.color`: uGUI `Button` owns that Image via its ColorTint transition and would fight per-frame writes. Hence the dedicated overlay Image in §4.
- **Guard**: `SelectedSlotIndex` is `-1` when nothing was selected (`SkillContracts.cs:319`) — compare `>= 0` explicitly, never a truthiness/`> 0` check (slot 0 is a real, and the most common, slot).

### 6. Per-selection sequence id for the pulse

**Files**: `Assets/Scripts/Data/State/SkillContracts.cs`, `Assets/Scripts/Core/Combat/ActiveSim.cs`

**Current state**: `AutoSkillSelectionDiagnostics` (`SkillContracts.cs:316-341`) carries `SelectedSlotIndex` (line 319, defaulting to `-1`), `SelectedSkillId`, a `Slots` list, and a `Clone()` (line 323). `ActiveSim` constructs a **fresh** diagnostics object per auto-rotation tick (`AutoCombatPolicy.cs:33`) and assigns it at `ActiveSim.cs:107-108`, *before* calling `ExecuteSkillAction` at line 111. A per-action counter already exists: `CombatSkillRuntimeState.LastActionSequenceId` (`SkillContracts.cs:256`), incremented by `checked(++...)` at `ActiveSim.cs:333, 400, 837, 1198`.

**Modifications**:
- `SkillContracts.cs`: add `public long SelectionSequenceId;` to `AutoSkillSelectionDiagnostics`, and copy it in `Clone()` (line 325-327) alongside `SelectedSlotIndex`. Missing it in `Clone()` would silently zero the field wherever combat state is cloned.
- `ActiveSim.cs`: in the auto-rotation branch, **after** `ExecuteSkillAction` (line 111) has incremented `LastActionSequenceId`, and only on the `skill != null` path, stamp `state.SkillRuntime.LastAutoSelection.SelectionSequenceId = state.SkillRuntime.LastActionSequenceId;`.
- **Ordering matters**: stamping before `ExecuteSkillAction` would record the *previous* action's id and the pulse would lag one cast behind.
- **Only on `skill != null`**: when auto-combat falls through to a basic attack (`ResolvePlayerAttack`, line 113), `LastAutoSelection` still holds the *previous* selection's stale `SelectedSlotIndex`. Stamping there would pulse a slot that did not fire. Leaving the id unchanged on that path suppresses the false pulse for free.
- This field is diagnostic output only — nothing in the simulation reads it, so determinism and offline math are unaffected. It is not persisted (combat state is runtime-only), so there is no save impact.

## Technical Considerations

- **Pattern Usage**: Highlight reuses `SkillBarPanel`'s established transient-timer idiom (`_feedbackUntil`) rather than adding a coroutine or Animator. Slot widgets follow the existing `BuildSkillBar` construction pattern.
- **Layer Placement**: Slot constant + `SelectionSequenceId` field → Data (`IdleCloud.Data`, engine-free — an `int` const and a `long`). Scan-limit fix + sequence stamp → Core (`IdleCloud.Core`, engine-free). Hotkeys, overlay, theme constants → `IdleCloud.UI`. **No new asmdef references.** The highlight reads `ActiveCombatState` through `GameManager`, the same path `SkillBarPanel` already uses — no upward dependency introduced.
- **Determinism**: Untouched. `AutoCombatPolicy` gains no randomness; widening the scan range only lets it see slots that were already in the persisted state. `SelectionSequenceId` derives from the existing deterministic `LastActionSequenceId` counter and is diagnostic output — nothing in the simulation branches on it.
- **Editor Authoring Impact**: `slotAutoHighlights` is `[HideInInspector]` and populated by `UIBuilder`, matching the other slot arrays — no manual Inspector wiring. **A UI rebake is required** (`IdleCloud > UI > Bake UI Prefab`) because slot count and per-slot hierarchy both change; the baked `GameUI.prefab` is authoritative. Rebake **with `PersistentGame.unity` as the active scene**, then save the scene.
- **Save Compatibility**: **No schema version bump.** `NormalizeSkillBars` (`SaveManager.cs:230-254`) unconditionally rebuilds every bar to exactly `Character.SkillBarSlots` entries on every load, so a 4-entry save pads to 8 with `null` and keeps its 4 assignments in place. The reverse (an 8-entry bar loaded by a 4-slot build) would silently truncate slots 5–8 — acceptable pre-release, but worth stating: this change is **not** backward-downgradable.
- **Edge Cases**:
  - **Free-grant decoupling (discovered during implementation — the plan's original assumption was wrong).** `CreateCharacter` seeds `UnlockedSkillIds` from the default bar (`Character.cs:56,75`), so *every skill placed in the default bar is granted for free*. The plan assumed classes define ~4 skills; under the production `SkillContentRegistry` a class has more, so seeding `SkillBarSlots` skills would have auto-unlocked nearly all of them and bypassed skill-point progression (caught by `SkillUnlockSlotTests` failing with `skill_not_unlockable`). Fixed by `Character.DefaultSeededSkillBarSlots = 4`: bar *length* is 8, but only the leading 4 slots are pre-filled/granted, exactly as before.
  - Classes currently define ~4 skills, so slots 5–8 sit empty on day one. Empty slots render `"-"`, are non-interactable (`Refresh`, line 219), and auto-combat records `empty_slot` and moves on (`AutoCombatPolicy.cs:46-50`) — all pre-existing behavior.
  - Hotkeys 5–8 on an empty slot: `TriggerSkillSlot` returns early at the null-skill check (line 91) — silent no-op, no feedback flash.
  - `LastAutoSelection` is `null` before the first combat tick and on a fresh `ActiveCombatState` — null-guard the whole chain.
  - **Auto-combat repeatedly casting from the same slot** (the common case — slot 0 wins most ticks) must pulse on *every* cast, not just the first. This is exactly why the trigger keys on `SelectionSequenceId` rather than on a change in `SelectedSlotIndex`; see §6.
  - Auto-combat falling through to a basic attack leaves a stale `SelectedSlotIndex` behind — must not pulse (handled by stamping only on the `skill != null` path, §6).
  - Manual triggers must **not** raise the highlight — it indicates auto-combat's choice specifically; the existing `"QUEUED"` / `"CAST!"` feedback already covers manual casts, and manual casts never reach the auto-rotation stamp.
  - `SelectionSequenceId` starts at `0`, which is also `_lastAutoSelectionSequenceId`'s initial value — the first auto cast stamps a value `>= 1` (the counter pre-increments), so no pulse is lost and none fires spuriously at boot.

## Type Definitions

One added field — no new types:

- `AutoSkillSelectionDiagnostics.SelectionSequenceId` (`long`, `SkillContracts.cs`): the value of `CombatSkillRuntimeState.LastActionSequenceId` at the moment of the auto cast this diagnostics object describes. `0` = no auto cast recorded yet. Diagnostic-only; must be copied in `Clone()`.

## Performance & Cost Impact

Negligible. Auto-combat's scan loop goes from ≤4 to ≤8 iterations per tick, short-circuiting on the first eligible slot as before — in practice unchanged, since classes have ~4 skills. The highlight adds one alpha write per slot per frame (8 writes), alongside the cooldown-fill refresh already running at that cadence. Four extra Images in the baked prefab.

## Backward Compatibility

Forward-compatible with no migration step: existing 4-entry saves pad to 8 via the existing normalization pass, preserving assignments. No `SaveSchemaVersion` bump. Downgrading a save written by this build back to a 4-slot build would truncate slots 5–8 — flagged, not mitigated (pre-release).
