# Flow A: Fixed-Tick Active Combat + Grid-Based Nav Queries — Implementation Plan

## Overview

Align the active combat loop with the Flow A contract (`### Flow A Active Combat Loop.md`, PROJECT.md §4 Flow A): the combat simulation must run on a **deterministic fixed tick** decoupled from Unity's visual framerate, and spatial combat facts (range, line of sight, nearest-target distance) must come from the **2.5D grid** (`GridPathfinder` / `TerrainHeightService`) instead of frame-dependent Physics2D queries and world-unit float distances.

Scope decisions (user-confirmed 2026-07-16):
- **In scope:** fixed tick + grid-aware nav queries.
- **Out of scope (deferred):** physical loot-drop entities when Auto-Loot is off (loot keeps flowing directly to inventory/bank).
- **Movement stays in View:** Core continues to emit `MovementRequested` intents; `PlayerController`/`GridPathfinder` execute them. The Flow A doc's "ActiveSim processes movement" wording is treated as aspirational — the engine-free Core wall wins.
- **Tick driver:** time accumulator inside `CombatView.Update()` (no new GameObjects, no FixedUpdate coupling).

## Problem Statement

1. `CombatView.Update()` calls `GameManager.TickActiveCombat(...)` once per rendered frame — tick cadence varies with framerate (30 fps machine ticks 3× slower than 144 fps in wall-clock density; the sim self-corrects via timestamps, but command latency, event burst shape, and attack-resolution granularity are all frame-dependent). The Flow A doc requires a fixed tick (10–20/s).
2. `GameManager.TickActiveCombat` stamps every tick with `DateTimeOffset.UtcNow` internally — callers cannot supply fixed-step timestamps, so two catch-up ticks in one frame would collapse to (nearly) the same timestamp.
3. Range is `Vector2.Distance(player, target) <= attackRange` in world units — on an isometric grid (cell 0.64×0.32) a world-unit radius is anisotropic (covers more cells along X than Y) and ignores height entirely: a player can "melee" a target one cell away but 3 height levels up a cliff.
4. Line of sight is `Physics2D.Linecast` against `obstacleLayers` — physics state, frame-timing dependent, and blind to the terrain height model that the rest of the project (pathfinding, sorting) is built on.
5. Auto-target selection feeds `AutoCombatPolicy.SelectTarget` with float world distances — works, but inconsistent with the grid model and subject to float ties.

## Solution Architecture

Two independent work packages, both View/Managers-layer only — **no Core or Data changes**:

**A. Grid-based combat world facts.** `GridPathfinder` gains a small public spatial-query API (world→cell, cell distance, height-aware line of sight). `CombatView` uses it to build `CombatWorldFacts`: range becomes "Chebyshev cell distance ≤ N cells AND |height delta| ≤ M levels", LoS becomes a grid line-walk blocked by intermediate cells taller than both endpoints, and auto-target candidate distances become cell distances. `Physics2D.Linecast` and `obstacleLayers` are removed from combat. When the target comes into range, `CombatView` cancels the player's in-progress path (new tiny `PlayerController` public method) so the player stops at attack range instead of walking into the mob.

**B. Fixed-tick loop.** `CombatView.Update()` accumulates `Time.deltaTime` and runs 0..N fixed sim steps per frame at a serialized tick rate (default 20/s — within the doc's 10–20 band, keeps manual-command latency ≤ 50 ms so clicks still feel instant). A sim clock (anchored to wall clock, advanced in fixed `tickMs` steps) supplies the timestamp; `GameManager.TickActiveCombat` gains a `long timestampMs` parameter and stops reading `DateTimeOffset.UtcNow` for tick time. Catch-up is capped per frame; after a long stall (alt-tab, editor pause) the clock re-anchors forward to wall clock rather than grinding hundreds of ticks — `ActiveSim` already tolerates forward timestamp jumps natively (all its scheduling is `Timestamp >= nextAt` comparisons; it only rejects backwards movement).

Rendering/feedback is untouched: `CombatFeedbackView` and UI already consume `ActiveCombatResolved` events and render on their own frames — exactly the doc's §6 split.

## Implementation Details

### 1. GridPathfinder — public spatial-query API

**File**: `Assets/Scripts/View/GridPathfinder.cs`

**Current state:** exposes `FindPath`, `IsWalkable`, `TryGetHeightAt`, plus private `ToCellXY`, `GetHeight`, `IsStandable`. No public cell-distance or LoS query.

**Modifications**:
- New `public bool TryGetCell(Vector3 world, out Vector2Int cell)` — exposes the existing private `ToCellXY` conversion (guarding the no-`_grid` case, same convention as `TryGetHeightAt` around line 122).
- New `public int CellDistance(Vector3 a, Vector3 b)` — Chebyshev distance (`max(|dx|,|dy|)`) between the two world positions' cells; returns a sentinel (e.g. `int.MaxValue`) when no grid exists. Chebyshev matches `Neighbors8`'s 8-directional adjacency: distance 1 == "one step away, diagonals included".
- New `public bool HasLineOfSight(Vector3 from, Vector3 to)` — integer line-walk (Bresenham/supercover) over the cells strictly between the two endpoint cells; blocked if any intermediate cell's height (`GetHeight`, around line 100) exceeds `max(fromHeight, toHeight)`. Endpoint cells excluded (you can always see the cell you and the target stand on). Adjacent cells (Chebyshev ≤ 1) have no intermediate cells → trivially true. No grid/height data → returns true (fail-soft, matches §17 View-layer convention).
- No change to `FindPath`, `IsStandable`, climb logic, or any existing member — additive only.

### 2. CombatView — grid facts + fixed-tick accumulator

**File**: `Assets/Scripts/View/CombatView.cs`

**Current state:** `Update()` ticks the sim once per frame with `Vector2.Distance` range (`attackRange = 1.1f` world units, line 11), `Physics2D.Linecast` LoS (lines 88–89), and float world distances for auto-target candidates (line 164). Handles `MovementRequested` events by calling `_player.RequestMoveToWorld` (line 107).

**Modifications**:
- **Serialized fields:** remove `attackRange` (float) and `obstacleLayers` (LayerMask). Add `public GridPathfinder pathfinder` (public field, mirroring `PlayerController.pathfinder`/`EnemyController.pathfinder` exactly — that is the convention `SceneBootstrap`'s wiring block already assigns through), `attackRangeCells` (int, default 1, min 1), `maxAttackHeightDelta` (int, default 1 — mirrors `maxHeightStepPerMove`), `ticksPerSecond` (float, default 20, range ~5–30), `maxCatchUpTicksPerFrame` (int, default 5). All tunable in the Inspector per project rule 2.
- **Pathfinder resolution (Start-order safe):** `GridPathfinder` does not exist until `SceneBootstrap.Start()` creates it via `AddComponent`, and Unity does not guarantee `Start()` order — a one-shot `FindFirstObjectByType` in `CombatView.Start()` can permanently miss it (exact bug class fixed for `EnemyController` on 2026-07-12, see STATE.md). Apply the same two-part fix: (a) `SceneBootstrap` wires the created pathfinder onto the `CombatView` right after the existing player/enemy wiring (deterministic), and (b) `CombatView` lazily re-resolves via `FindFirstObjectByType` at tick time while its reference is still null (self-heal). The permissive gridless fallback only applies when resolution genuinely fails (no pathfinder in scene at all).
- **World facts (per tick, not per frame):** in-range = `pathfinder.CellDistance(player._logical→position, target.position) <= attackRangeCells` AND `|heightAt(player) − heightAt(target)| <= maxAttackHeightDelta` (heights via `TryGetHeightAt`); LoS = `pathfinder.HasLineOfSight(...)`; `CombatWorldFacts.Distance` = the cell distance (existing `double` field reused, no Data change). If `pathfinder` is null (no grid scene), fall back to the always-true/available facts — combat still functions in gridless test scenes.
- **Auto-target selection:** `SelectAutoTarget()` fills `CombatCandidate.Distance` with cell distance instead of `Vector2.Distance`. `AutoCombatPolicy.SelectTarget`'s existing ordinal-ID tie-break now does real work (integer distances tie often) — deterministic selection, no Core change. **Gridless fallback (defined):** when the pathfinder is null or `TryGetCell` fails for either endpoint, the candidate distance falls back to `Vector2.Distance` in world units — exactly today's behavior, still deterministic for a fixed scene layout.
- **Stop at attack range (chase paths only):** `CombatView` tracks *chase-path ownership*: a private flag set when it issues `RequestMoveToWorld` from the `MovementRequested`/out-of-range branch, cleared by the existing `ManualMoveRequested` handler (already subscribed), on target loss, immediately after `CancelPath()` consumes the chase path, and whenever combat stops or a tick is rejected — so a later path issued by gathering or any other system can never be mistaken for a stale chase. `PlayerController.CancelPath()` is called only when the flag is set and the tick's facts say in-range — a manual click-to-move (e.g. the player walking away from an adjacent mob) is never cancelled. The out-of-range branch itself (`MovementRequested` → `RequestMoveToWorld(target.position)`) is unchanged apart from setting the flag.
- **Fixed-tick loop:** `Update()` becomes: (1) resolve manager/player/target exactly as today (target acquisition and `StartActiveCombat` stay frame-driven — they're cheap and input-latency-sensitive); (2) **drift check first:** compare `_simTimeMs` against `DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()`; if wall clock is ahead by more than `maxCatchUpTicksPerFrame × tickMs`, re-anchor — jump `_simTimeMs` forward to wall clock, reset the accumulator, and **end tick processing for this frame** (no accumulation, no ticks — otherwise the just-anchored clock would immediately advance up to `maxCatchUpTicksPerFrame × tickMs` ahead of wall time; normal ticking resumes next frame). This must be wall-clock-based, **not** accumulator-based: Unity clamps `Time.deltaTime` to `maximumDeltaTime` (0.333 s in this project's `TimeManager.asset`), so a long alt-tab/pause never shows up in accumulated delta time. (3) accumulate `Time.deltaTime`; (4) while accumulated ≥ tick interval and ticks-this-frame < `maxCatchUpTicksPerFrame`: advance the sim clock by `tickMs`, build world facts, call `manager.TickActiveCombat(..., simTimeMs)`, process events per tick as the current per-frame body does — **and if the tick resolved a reward or player defeat (terminal tick), break out of the catch-up loop immediately and reset the accumulator**: the current body nulls `_target`/resets `_combatStarted` on those outcomes, so a further queued tick this frame would dereference a cleared target; the next frame re-acquires cleanly. Sim clock: `_simTimeMs` initialized from wall clock on first tick; strictly monotonic (fixed steps + forward-only re-anchor) — `ActiveSim`'s `timestamp_moved_backwards` guard can never fire.
- **No tick while no target/combat:** the accumulator only drains while a started combat exists (same guard positions as the current early-returns), so idle scenes pay nothing.

## Technical Considerations

- **Pattern Usage:** `pathfinder` auto-resolve mirrors `PlayerController`/`EnemyController`; fail-soft null-grid behavior mirrors §17 (ARCHI) View conventions; public query methods follow `TryGetHeightAt`'s existing shape; `TerrainHeightService` remains the single height authority (no second height map — hard constraint from STATE.md history).
- **Layer Placement:** all changes in `IdleCloud.View` (CombatView, GridPathfinder, PlayerController), `IdleCloud.Managers` (GameManager), `IdleCloud.Tests.PlayMode`. **Zero Core/Data edits** — `CombatWorldFacts`, `ActiveSim`, `AutoCombatPolicy` are consumed as-is. Dependency flow stays downward.
- **Determinism:** fixed tick makes cadence framerate-independent; integer cell distances + ordinal tie-break make auto-targeting deterministic. Active combat keeps `SystemRandomSource` (existing, unseeded) — the seeded-randomness requirement applies to offline (Flow C), which is untouched; offline math stays rate-based.
- **Editor Authoring Impact:** no UI rebake (no `Assets/Scripts/UI` change), no prefab regeneration. **Scene edit required by user:** the `CombatView` component instance (in the map/persistent scene) loses `attackRange`/`obstacleLayers` and gains 5 new serialized fields — user must select it in the live Editor, confirm defaults / drag the `GridPathfinder` reference (optional, auto-resolves), and save the scene so orphaned serialized values are dropped.
- **Save Compatibility:** none — no snapshot/state shape change (`ActiveCombatState` untouched; it is transient, never serialized to saves anyway).
- **Edge Cases:**
  - *Catch-up spiral:* `maxCatchUpTicksPerFrame` caps per-frame work; long stalls are detected by wall-clock drift (`_simTimeMs` vs `UtcNow`), never by accumulator size — Unity's `maximumDeltaTime` clamp (0.333 s) hides stalls from `Time.deltaTime`.
  - *Terminal tick mid-catch-up:* reward/defeat breaks the tick loop for that frame (target/state already cleared); accumulator resets, next frame re-acquires.
  - *Timestamp monotonicity:* fixed steps + forward-only re-anchor ⇒ `ActiveSim`'s backwards-timestamp rejection is unreachable from this driver.
  - *Manual escape while adjacent:* chase-ownership flag ensures stop-at-range never cancels a manual click-to-move away from the mob.
  - *Range-boundary oscillation:* discrete cell distances can't flutter the way float world distances at a threshold can; a wandering target flips range by whole cells only.
  - *Cliff-edge melee:* adjacent cell but height delta 2+ → out of range → `MovementRequested`, player paths around via climb moves (existing `GridPathfinder` behavior).
  - *LoS across a ridge:* intermediate cell taller than both endpoints blocks; endpoints and adjacent-melee always pass.
  - *No grid / legacy sandbox scenes:* pathfinder-null fallback keeps combat functional (facts default permissive).
  - *Command latency:* UI skill buttons / manual clicks enqueue into `GameManager._queuedCombatCommands` (existing buffer); at 20 tps worst-case latency is 50 ms — imperceptible, and the buffer already guarantees no command is lost between ticks.

## Type Definitions

No new types. No changes to `CombatWorldFacts`, `ActiveCombatState`, `ActiveCombatConfig`, or any Data/Core contract. New public members only: three `GridPathfinder` query methods, one `PlayerController` method (`CancelPath`), one public `CombatView.pathfinder` field (wiring convention shared with `PlayerController`/`EnemyController`), one added `GameManager.TickActiveCombat` parameter.

## Performance & Cost Impact

- Sim work moves from once-per-frame (up to 144+/s) to fixed 20/s — a net *reduction* on high-refresh machines; catch-up bounded at 5 ticks/frame.
- `CellDistance`/`HasLineOfSight` are O(1) / O(line length in cells) dictionary lookups per tick — negligible against the existing per-frame `Physics2D.Linecast` they replace.
- Auto-target scan stays O(targets) per frame as today (target acquisition remains frame-driven for input responsiveness).

## Backward Compatibility

- No save/snapshot impact.
- Scene-serialized `attackRange`/`obstacleLayers` values on existing `CombatView` instances become orphaned data; Unity drops them silently on next scene save (same pattern as the 2026-07-12 SceneBootstrap field removals). Combat behavior change is intentional: range semantics switch from 1.1 world units to 1 cell (Chebyshev) — on this grid that is a slight *tightening* along X and a *loosening* along Y; user validates feel in Play mode and tunes `attackRangeCells` if needed.

## Test Impact

- `GameplayLoopSmokeTests` (PlayMode): compile-breaks on the `TickActiveCombat` signature change — call-site updated to pass a timestamp; behavior assertions unchanged. Remains View-referencing, **no UI reference added** (hard rule from ARCHI §14).
- Existing EditMode suites (ActiveSim, AutoCombatPolicy, coordinator, drops, save-compat): unaffected — zero Core/Data/coordinator changes.
- New grid-query logic (`CellDistance`, `HasLineOfSight`) lives in `IdleCloud.View`, which the EditMode test asmdef does not reference — coverage lands on the PlayMode smoke test plus manual Play-mode verification (TRIP-2 gate decides whether to extend `GameplayLoopSmokeTests` with a LoS/range scenario).
- Manual Play-mode verification (user, live Editor) is the primary gate for: stop-at-range feel, cliff-edge out-of-range pathing, LoS blocking across a ridge, tick-rate feel at 20 tps, and death/respawn timing under the sim clock.
